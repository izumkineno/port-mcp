# 20 M4 队列缓冲区与订阅施工记录

本文记录 M4 阶段在 `runtime` 内落地的 Mock-only 队列、接收缓冲区、订阅者队列、动态预算和通知限速能力。M4 继续遵守 M3 之后的边界：不修改 `mcp-server/`，不实现真实 Serial/TCP/UDP I/O，不实现 `port_scan`，不接入 MCP 全工具 handler。

## 阶段范围

| 范围 | 结果 | 说明 |
| --- | --- | --- |
| 有界 tx 队列 | 已完成 | `port_send_mock` 按单帧上限和队列项上限入队，`flush_mock_tx` FIFO 消费并归还预算。 |
| rx buffer | 已完成 | `receive_mock_chunk` 写入普通 rx buffer，满时丢旧保新，`port_pull_mock` 消耗式读取并返回 truncated 与剩余字节数。 |
| `port_clear` | 已完成 | `port_clear_mock` 支持 `Tx`、`Rx`、`All`，清理运行时队列和 buffer，不撤回已广播到订阅者队列的通知。 |
| 订阅者队列 | 已完成 | `subscribe_mock`、`pull_subscription_mock`、`unsubscribe_mock` 支持每 session 独立队列、重复取消幂等和订阅上限。 |
| 慢订阅者隔离 | 已完成 | 订阅者队列满时只丢弃该订阅者旧通知，不影响普通 rx buffer 或其他订阅者。 |
| 动态预算扣账 | 已完成 | tx 队列、订阅队列和 rx buffer 在入队/拉取/clear/disconnect/release 路径归还预算。 |
| 通知限速 | 已完成 | 按单订阅者、单实例和全局 tick 计数限速，限速路径保留最新通知并标记 coalesced。 |
| Mock 全生命周期 | 已完成 | Mock 路径覆盖 send、pull、clear、subscribe、task error 和 force release。 |

## 代码落点

| 文件 | 变更 |
| --- | --- |
| `src/runtime/mod.rs` | 扩展 `RuntimeRegistry` 的 `RuntimeLimits`、rx/tx 预算、全局通知 tick；扩展 `RuntimeInstance` 的 tx queue、rx buffer、subscribers 和通知计数；新增 Mock-only port API 与 M4 结果类型。 |
| `src/model/mod.rs` | 沿用 M1 已定义的 `RuntimeLimits`、`InstanceStats` 和 M4 错误码；本阶段未新增模型文件。 |
| `src/transport/mod.rs` | 保持 M3 MockTransport 边界；M4 不引入真实 I/O。 |

M4 没有为了贴合 [04-运行时架构设计](04-运行时架构设计.md) 进行纯空文件拆分。队列、缓冲区和订阅仍集中在 `runtime` 中，因为当前能力仍是 Mock-only 运行时模型；后续 M5/M7 接入真实 transport 或 MCP handler 时，可在保持测试不变的前提下拆出 `runtime::queues`、`runtime::buffers`、`runtime::subscription` 等子模块。

## 边界确认

- 未进入 M5/M6/M7/M8。
- 未实现真实 TCP、UDP 或 Serial I/O。
- 未实现 `port_scan`、串口扫描、MCP handler 全工具接入或发布前验收。
- 后台任务和 transport 仍只通过 MockTransport 与无真实 I/O 的可测试模型表达。
- `mcp-server/` 未修改。
