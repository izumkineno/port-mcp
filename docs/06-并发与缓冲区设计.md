# 06 端口 MCP 并发与缓冲区设计

本文档承接 [01-初版工程功能设计大纲](01-初版工程功能设计大纲.md)、[03-初版工具契约](03-初版工具契约.md)、[04-运行时架构设计](04-运行时架构设计.md) 与 [05-状态机与资源生命周期](05-状态机与资源生命周期.md)，固化初版发送队列、普通 `rx` 缓冲区、订阅广播、多 AI 会话隔离、容量上限、溢出策略和并发调用一致性。

本文档只定义运行时内存并发语义。具体错误码字段、日志字段和用户可读恢复建议由 `07-错误模型与返回格式.md` 承接；测试矩阵由 `08-测试与验收计划.md` 承接。

## 设计结论

- 每个实例拥有独立发送队列、普通 `rx` 缓冲区和订阅者集合，实例之间互不共享数据路径。
- `port_send` 只负责校验、编码和入队；底层写入由每实例发送任务按入队顺序串行执行。
- 发送队列满时拒绝新的 `port_send`，返回 `BufferLimitExceeded` 或后续 `07` 定义的等价错误，不丢弃已排队数据。
- 接收任务读取到底层数据后，先写入普通 `rx` 缓冲区，再广播给订阅者。
- 普通 `rx` 缓冲区满时采用丢旧保新策略，并累计 `rx_dropped_bytes`。
- 每个订阅者拥有独立通知队列；慢订阅者只丢弃自己的旧通知并计数，不阻塞接收任务和其他订阅者。
- `port_pull` 是普通 `rx` 缓冲区的消耗式读取，不影响已广播或已排队给订阅者的数据。
- `port_clear(rx)` 只清普通 `rx` 缓冲区；`port_clear(tx)` 只清尚未写入底层资源的发送队列项；二者都不撤回已经投递给订阅者的通知。
- 断开连接会关闭发送入口、丢弃未写入发送项、停止订阅广播并保留普通 `rx` 缓冲区；释放实例会清空全部队列、缓冲区和订阅记录。

## 推荐默认限制

初版建议在 `RuntimeLimits` 中集中定义以下默认值，后续可由运行配置覆盖。

| 限制项 | 建议默认值 | 说明 |
| --- | --- | --- |
| `tx_queue_max_items` | `256` | 每实例最多等待写入的发送项数量。 |
| `tx_frame_max_bytes` | `64 KiB` | 单次 `port_send` 编码后的最大字节数。 |
| `rx_buffer_max_bytes` | `1 MiB` | 每实例普通 `rx` 缓冲区最大容量。 |
| `pull_default_max_bytes` | `4 KiB` | `port_pull` 未提供 `max_bytes` 时的默认返回上限。 |
| `pull_max_bytes` | `64 KiB` | 单次 `port_pull` 允许返回的最大字节数。 |
| `subscriber_queue_max_items` | `128` | 每个订阅者最多积压的通知数量。 |
| `subscriber_payload_max_bytes` | `16 KiB` | 单次订阅通知 payload 最大字节数。 |
| `scan_max_concurrency` | `64` | 网络扫描的最大并发探测数。 |
| `scan_max_ports` | `256` | 单次网络扫描允许探测的最大端口数。 |
| `scan_total_timeout_ms` | `10000` | 单次网络扫描整体超时上限。 |
| `max_instances` | `64` | 单进程最多未释放实例数。 |
| `max_subscribers_per_instance` | `32` | 单实例最多订阅者数量。 |
| `max_total_buffer_bytes` | `128 MiB` | 全进程普通 `rx` 缓冲区预算硬上限。 |
| `max_total_queued_bytes` | `64 MiB` | 全进程发送队列与订阅队列估算预算硬上限。 |

这些值不是协议永久承诺，但初版实现和测试应围绕它们建立默认行为。若后续配置允许修改，也必须保留全局绝对硬上限，至少包括 `ABS_MAX_TOTAL_BUFFER_BYTES` 和 `ABS_MAX_TOTAL_QUEUED_BYTES`，避免 AI 调用一次性占用过多内存。

初版采用动态预算扣账，而不是按所有实例的最坏情况静态预留内存。运行时必须维护至少两个全局预算计数器：

| 预算 | 计入对象 | 失败行为 |
| --- | --- | --- |
| `buffer_bytes_budget` | 普通 `rx_buffer` 当前实际持有字节。 | append 前无法 reserve 时按 rx 丢旧保新策略释放旧字节；仍无法满足时丢弃并累计统计。 |
| `queued_bytes_budget` | 已入队但未释放的 tx frame 字节、订阅者队列 payload 字节。 | `port_send` 或订阅通知入队前无法 reserve 时，返回容量错误或按慢订阅者丢旧策略释放旧通知。 |

每次 tx item 入队、订阅通知入队、rx append 前必须先 reserve 对应预算；发送完成、队列丢弃、订阅通知丢弃、`port_clear`、disconnect 或 release 清理时必须归还预算。默认值允许大量实例存在，但实际内存占用由动态预算限制；测试应覆盖 reserve 失败、丢弃后归还预算和长时间运行预算不泄漏。

## 并发对象模型

每个 `RuntimeInstance` 至少包含以下并发对象：

```text
RuntimeInstance
- state
- stats
- tx_queue
- rx_buffer
- subscribers
- task_group
- cancellation_token
```

推荐职责如下：

| 对象 | 写入方 | 读取方 | 并发要求 |
| --- | --- | --- | --- |
| `tx_queue` | `port_send`、`port_clear(tx)`、断开/释放清理 | 发送任务 | 多生产者、单消费者，保持 FIFO。 |
| `rx_buffer` | 接收任务、`port_clear(rx)`、释放清理 | `port_pull`、`instance_query` | 单生产者、多工具消费者，读取必须消耗。 |
| `subscribers` | `port_subscribe_stream`、`port_unsubscribe_stream`、断开/释放清理 | 接收任务广播 | 订阅表变更不能阻塞接收 I/O。 |
| `stats` | 发送任务、接收任务、清理流程 | `instance_query` | 计数更新应原子或短锁保护。 |
| `task_group` | `port_connect`、断开/释放清理 | 状态机清理流程 | 任务退出原因必须可收集。 |

工具调用不得在持有实例状态锁时执行长时间 I/O、等待订阅者消费、等待队列空闲或执行底层 transport 读写。

## 发送队列

### 入队流程

```text
port_send
-> check state == Connected
-> decode text/hex into bytes
-> apply append_line_break if needed
-> validate frame bytes <= tx_frame_max_bytes
-> try enqueue SendItem into tx_queue before timeout
-> return queued=true and sent_bytes=frame_len
```

`sent_bytes` 表示运行时已经接收的待发送字节数，不保证底层设备已经完成物理写入。发送任务后续写入失败时，实例按 [05-状态机与资源生命周期](05-状态机与资源生命周期.md) 进入 `Error`。

### 顺序保证

- 同一实例内，发送任务必须按 `tx_queue` 入队顺序写入底层资源。
- 单个 `port_send` 编码后的字节帧必须作为不可拆分的发送项处理，不允许与其他发送项交织。
- 不同实例的发送队列互不影响，可以并发写入各自底层资源。
- 多个 AI 会话同时调用同一实例的 `port_send` 时，只承诺按运行时成功入队顺序写入，不承诺按请求到达网络或 UI 的先后顺序写入。

### 队列满策略

发送队列满时采用拒绝新项策略：

```text
tx_queue full
-> reject current port_send
-> keep existing queued items
-> increment tx_rejected_items
-> return BufferLimitExceeded
```

拒绝新项比丢弃旧项更适合命令式端口通信：已经接受的命令不能被静默移除，新命令失败也更容易被 AI 重试或降速。

### 清理发送队列

`port_clear(buffer_type=tx)` 只删除尚未被发送任务取出的项：

```text
port_clear(tx)
-> lock or swap tx_queue pending segment
-> drop pending SendItem values
-> increment tx_dropped_items
-> return dropped_tx_items
```

已经被发送任务取出并正在写入底层资源的帧不可撤回。断开连接时，发送入口关闭，未写入项丢弃并计入清理结果或日志。

## 普通 `rx` 缓冲区

### 写入流程

接收任务从 transport 得到数据后：

```text
receive task
-> read chunk from transport
-> update rx_bytes and last_activity_at
-> append chunk into rx_buffer with overflow policy
-> broadcast chunk to subscribers
```

普通 `rx` 缓冲区是实例级共享缓冲区，不按 AI 会话拆分。任何会话对同一实例调用 `port_pull` 都会消费同一个普通 `rx` 缓冲区。

UDP 初版在普通 `rx` 缓冲区中采用字节流语义：每个 datagram 的 payload 按接收顺序追加到同一个 `rx_buffer`，不在 `port_pull` 返回中保留 datagram 边界或来源地址。需要完整 datagram 帧队列、来源地址元数据或多来源区分时，应扩展进阶契约，不能在初版实现中隐式改变 `port_pull` 外形。

### 溢出策略

普通 `rx` 缓冲区满时采用丢旧保新策略：

```text
append chunk
-> if chunk.len > rx_buffer_max_bytes:
     keep newest rx_buffer_max_bytes bytes from chunk
     increment rx_dropped_bytes by dropped bytes
-> else if rx_buffer.len + chunk.len > rx_buffer_max_bytes:
     drop oldest bytes until new chunk fits
     append chunk
     increment rx_dropped_bytes
-> else append chunk
```

接收溢出不直接使实例进入 `Error`。`instance_query` 应能暴露当前 `rx_buffer_bytes` 和累计 `rx_dropped_bytes`，让 AI 判断是否需要提高拉取频率、清理缓冲区或降低设备输出速率。

### 拉取流程

```text
port_pull
-> check state == Connected
-> wait until rx_buffer has data or timeout_ms expires
-> take up to min(request.max_bytes, pull_max_bytes)
-> remove returned bytes from rx_buffer
-> encode payload as hex/text
-> return remaining_rx_buffer_bytes and truncated flag
```

`truncated=true` 表示本次只返回了缓冲区中的一部分数据，剩余字节仍保留在普通 `rx` 缓冲区。超时无数据返回 `ReadTimeout`，不改变实例状态。

### 清理接收缓冲区

`port_clear(buffer_type=rx)` 清空普通 `rx` 缓冲区，并返回被丢弃字节数。它不影响：

- 已经通过订阅广播发送出去的通知。
- 每个订阅者队列中尚未消费的通知。
- 累计 `rx_bytes` 和 `rx_dropped_bytes` 统计。

## 订阅广播

### 订阅模型

订阅关系由 `handle_id + session_id` 唯一确定。同一会话重复订阅同一实例应幂等成功，可以更新 `max_payload_bytes`，但不创建多个重复订阅者。

每个订阅者维护独立队列：

```text
Subscriber
- session_id
- stream_uri
- max_payload_bytes
- queue
- dropped_notifications
- dropped_bytes
- rate_limiter
```

### 广播流程

```text
receive chunk
-> snapshot current subscriber list
-> for each subscriber:
     split or truncate chunk to subscriber.max_payload_bytes
     try enqueue notification into subscriber.queue
     if subscriber.queue full:
       drop oldest notification for that subscriber
       enqueue newest notification
       increment subscriber dropped counters
```

广播不得等待慢订阅者消费。一个订阅者的队列满只影响该订阅者自己的通知完整性，不影响普通 `rx` 缓冲区、接收任务、其他订阅者或 `port_pull`。

对于 UDP，接收任务可以按单个 datagram 生成一次订阅通知，并在通知元数据中标记 `transport_protocol=udp`、`datagram=true`、`payload_bytes`。该元数据只用于订阅观察，不改变普通 `rx` 缓冲区的字节流读取语义。

### 通知频率保护

初版必须实现最小时间维度限速，避免高频设备或本地 loopback 输入造成通知风暴。限速是通知投递策略，不改变普通 `rx` 缓冲区内容，也不改变 `port_pull` 的消耗式读取语义。

初版至少维护以下 token bucket 或等价滑动窗口限制：

| 限制 | 默认值 | 作用 |
| --- | --- | --- |
| `subscriber_notifications_per_sec` | `64` | 单订阅者每秒可投递通知数。 |
| `instance_notifications_per_sec` | `256` | 单实例所有订阅者合计每秒可投递通知数。 |
| `global_notifications_per_sec` | `1024` | 全进程所有订阅者合计每秒可投递通知数。 |
| `notification_burst` | `128` | 每个 bucket 允许的短时突发容量。 |

当任一 bucket 超限时，初版采用丢弃旧通知并保留最新通知的策略：

- 若订阅者队列中已有待发送通知，优先合并为一条最新摘要通知，标记 `coalesced=true`、`dropped_notifications` 和 `latest_payload_bytes`。
- 若无法合并，丢弃当前通知，累计该订阅者和实例级 `dropped_notifications`。
- 限速丢弃必须使用限速日志，避免每个被丢弃通知都写一条 warn。
- 限速不得阻塞接收任务、发送任务、普通 `rx` 写入或其他未超限订阅者。

进阶版本可以增加 per-session 配额、自适应批处理、优先级和更复杂的 backpressure；这些不属于初版必需项。

### 通知 payload 限制

每次通知 payload 不得超过 `min(request.max_payload_bytes, subscriber_payload_max_bytes)`。当接收 chunk 超过该限制时，初版采用截断通知策略，并在通知元数据中标记 `truncated=true`、`original_bytes` 和 `payload_bytes`。

被截断的通知不影响普通 `rx` 缓冲区；完整数据仍按普通缓冲区策略保留，除非 `rx` 缓冲区自身发生丢旧溢出。

### 取消订阅与断开

| 事件 | 订阅队列处理 |
| --- | --- |
| `port_unsubscribe_stream` | 删除当前会话订阅者，丢弃其未消费通知。 |
| 同一会话重复取消 | 幂等成功，返回 `subscribed=false`。 |
| `port_disconnect` 成功 | 停止当前实例所有订阅者队列，订阅失效。 |
| 实例进入 `Error` | 停止当前实例所有订阅者队列，并记录错误结束原因。 |
| `instance_release` | 删除订阅表和所有订阅者队列。 |

## 多 AI 会话隔离

### 共享与隔离边界

| 内容 | 作用域 | 规则 |
| --- | --- | --- |
| 实例状态 | `handle_id` 共享 | 所有会话看到同一实例状态。 |
| 发送队列 | `handle_id` 共享 | 所有会话发送进入同一 FIFO 队列。 |
| 普通 `rx` 缓冲区 | `handle_id` 共享 | 任一会话 `port_pull` 都会消费共享缓冲区。 |
| 会话默认实例绑定 | `session_id` 独立 | `instance_use` 不影响其他会话。 |
| 订阅者队列 | `handle_id + session_id` 独立 | 慢订阅者只丢自己的通知。 |
| 订阅取消 | `session_id` 独立 | 一个会话取消不影响其他会话。 |

这意味着 `port_pull` 适合单消费者或协调好的问答流程；多个 AI 同时观察同一端口时，应优先使用订阅广播。

### 公平性

初版只保证同一实例内发送 FIFO 和订阅者互不阻塞，不保证跨会话公平调度。若某会话持续高速调用 `port_send`，其他会话的发送请求会排在其后。后续如需公平性，可在 `tx_queue` 前增加按会话分组的调度器，但不属于初版。

## `port_clear` 语义

| `buffer_type` | 清理对象 | 不清理对象 |
| --- | --- | --- |
| `tx` | 尚未被发送任务取出的发送项 | 正在写入的帧、普通 `rx`、订阅者队列 |
| `rx` | 普通 `rx` 缓冲区 | 发送队列、订阅者队列、已广播通知 |
| `all` | `tx` pending 项和普通 `rx` 缓冲区 | 正在写入的帧、订阅者队列、会话绑定 |

`port_clear` 不改变实例状态。对 `Error` 状态执行清理时，只清理仍存在的内存队列和缓冲区，不尝试重启任务或重新打开底层资源。

## 断开与释放时的缓冲处理

| 操作 | `tx_queue` | `rx_buffer` | 订阅者队列 |
| --- | --- | --- | --- |
| `port_disconnect` 成功 | 关闭入口，丢弃未写入项 | 保留 | 停止并失效 |
| `port_disconnect` 幂等成功 | 不改变 | 保留 | 不恢复订阅 |
| `port_disconnect` 失败进入 `Error` | 关闭入口或保持不可入队 | 保留 | 标记错误结束 |
| `instance_release` | 清空 | 清空 | 删除 |
| `instance_release(force=true)` | 清空或后台丢弃 | 清空 | 删除 |

断开保留普通 `rx` 缓冲区，是为了让用户在连接关闭后仍可通过 `instance_query` 看到缓冲长度，并在重新连接前用 `port_clear(rx)` 明确丢弃旧数据。由于 [05-状态机与资源生命周期](05-状态机与资源生命周期.md) 规定 `port_pull` 只允许 `Connected`，断开后的缓冲区不能继续被拉取。

## 统计字段建议

`instance_query` 已在 [03-初版工具契约](03-初版工具契约.md) 中定义基础字段。初版实现可额外暴露以下调试统计，具体返回结构由 `07` 固化：

| 字段 | 含义 |
| --- | --- |
| `tx_queue_items` | 当前等待写入的发送项数量。 |
| `tx_rejected_items` | 因发送队列满而拒绝的发送请求数。 |
| `tx_dropped_items` | 因 clear/disconnect/release 丢弃的待发送项数。 |
| `rx_buffer_bytes` | 当前普通 `rx` 缓冲区字节数。 |
| `rx_dropped_bytes` | 因普通 `rx` 缓冲区满而丢弃的旧字节数。 |
| `subscriber_count` | 当前订阅者数量。 |
| `subscriber_dropped_notifications` | 所有订阅者累计丢弃通知数量。 |
| `subscriber_dropped_bytes` | 所有订阅者累计丢弃通知字节数。 |

统计计数不得因为 `port_clear` 被重置。需要重置统计时，应由后续配置或调试工具另行定义。

## 实现建议

- `tx_queue` 可用 `tokio::sync::mpsc` 有界通道或等价有界队列实现。
- 普通 `rx_buffer` 可用 `VecDeque<u8>` 或环形缓冲区实现，必须支持高效丢弃旧字节。
- `port_pull` 等待数据可用 `Notify` 或条件变量等机制唤醒，避免轮询。
- 订阅者队列可用每订阅者一个有界队列，广播时对订阅表做快照，避免持锁发送通知。
- 统计字段可用原子计数或实例短锁更新；不要为了统计准确性阻塞 I/O 路径。
- 接收任务应先处理普通 `rx` 缓冲区，再处理订阅广播，保持 [01-初版工程功能设计大纲](01-初版工程功能设计大纲.md) 中的顺序语义。

## 设计验收标准

本文档完成后，初版工程应能据此实现和测试：

- 同一实例多会话并发 `port_send` 时，成功入队的数据帧按 FIFO 串行写入且不交织。
- 发送队列达到 `tx_queue_max_items` 后，新 `port_send` 被拒绝，已排队数据不丢失。
- 普通 `rx` 缓冲区达到 `rx_buffer_max_bytes` 后，旧字节被丢弃，新字节保留，并累计 `rx_dropped_bytes`。
- `port_pull` 从普通 `rx` 缓冲区消耗数据，并正确返回剩余长度和截断标记。
- 多个订阅者订阅同一实例时，都能收到同一接收数据；慢订阅者只丢自己的旧通知。
- 高频输入触发通知限速时，只影响通知投递，按 coalesce/drop 累计 `dropped_notifications`，不阻塞接收任务、普通 `rx` 或未超限订阅者。
- `port_clear(tx|rx|all)` 只清理指定内存对象，不撤回正在写入帧或已广播通知。
- 断开连接停止发送入口和订阅广播，释放实例清理全部队列、缓冲区、订阅和绑定。
