# 12 端口 MCP 调试记录格式

本文档承接 [07-错误模型与返回格式](07-错误模型与返回格式.md)、[08-测试与验收计划](08-测试与验收计划.md) 与 [11-配置与运行说明](11-配置与运行说明.md)，定义初版工程在真实设备调试、TCP/UDP loopback 验证、Windows 串口手工验收和问题复盘时应保存的记录结构。

本文档只定义基础框架和格式，不要求初版立刻实现自动导出工具。初版可以先由人工按模板记录；后续如实现自动导出，应保持字段语义与本文一致。

## 设计结论

- 调试记录以一次调试会话为单位，至少包含会话元数据、运行环境、目标资源、工具调用摘要、收发帧摘要、错误链、验收结论和已知限制。
- 记录格式优先使用 Markdown 作为人工可读报告，必要时附加 JSONL 事件流作为机器可分析原始记录。
- 记录必须能通过 `request_id`、`handle_id`、`last_error.error_id` 和日志事件关联工具返回、实例状态和后台任务。
- 所有记录默认脱敏，不保存系统用户名、完整本地路径、环境变量值、未裁剪 payload、认证信息或堆栈。
- 收发帧只保存长度、编码、短 preview、截断信息和方向；完整 payload 只有在明确安全且必要时才可单独保存。
- Windows 串口手工验收记录是初版发布前的重要证据；Linux/macOS 抽测记录作为已知限制和后续修复依据。

## 记录分层

| 层级 | 格式 | 用途 |
| --- | --- | --- |
| 调试报告 | Markdown | 人工阅读、验收归档、问题复盘。 |
| 事件流 | JSONL | 保存工具调用、日志事件、状态变化和收发摘要，便于后续过滤分析。 |
| 附件索引 | Markdown 表格或 JSON | 记录截图、外部串口工具日志、设备资料和复现实验脚本的位置。 |

初版可只要求 Markdown 调试报告。JSONL 事件流和附件索引属于增强项，但字段命名应从一开始保持稳定。

## 文件命名建议

调试记录文件建议按日期、平台和目标资源命名：

```text
debug-records/
  2026-05-25-windows-serial-com3-loopback.md
  2026-05-25-windows-serial-com3-loopback.events.jsonl
  2026-05-25-windows-serial-com3-loopback.attachments.md
```

命名规则：

| 片段 | 说明 |
| --- | --- |
| 日期 | 使用本地调试日期，格式 `YYYY-MM-DD`。 |
| 平台 | `windows`、`linux`、`macos` 或 `ci`。 |
| 资源类型 | `serial`、`tcp`、`udp` 或 `mock`。 |
| 资源摘要 | 例如 `com3`、`ttyusb0`、`tcp-loopback`、`udp-loopback`。 |
| 场景 | `loopback`、`permission-denied`、`busy`、`timeout`、`acceptance` 等。 |

调试记录是否提交到仓库由后续项目策略决定。若包含真实设备信息、客户环境信息或敏感日志，应只保存脱敏摘要。

## Markdown 调试报告模板

```markdown
# 调试记录：<场景标题>

## 基本信息

| 字段 | 值 |
| --- | --- |
| 日期 | 2026-05-25 |
| 记录人 | <姓名或代号> |
| 平台 | Windows / Linux / macOS / CI |
| 系统版本 | <脱敏版本摘要> |
| port-mcp 版本 | <commit / tag / build id> |
| MCP 客户端 | <客户端名称和版本> |
| 目标资源 | serial:COM3 / tcp:127.0.0.1:0 / udp:127.0.0.1:0 |
| 场景 | loopback / 权限错误 / 资源占用 / 超时 / 发布验收 |
| 结论 | 通过 / 不通过 / 未完成 / 需复测 |

## 运行配置

| 配置项 | 值 |
| --- | --- |
| log_level | info / debug |
| verbose_error_details | false |
| tx_queue_max_items | 256 |
| tx_frame_max_bytes | 64 KiB |
| rx_buffer_max_bytes | 1 MiB |
| pull_default_max_bytes | 4 KiB |
| pull_max_bytes | 64 KiB |
| subscriber_queue_max_items | 128 |
| subscriber_payload_max_bytes | 16 KiB |
| scan_max_concurrency | 64 |

## 设备与资源

| 字段 | 值 |
| --- | --- |
| 设备方案 | 真实设备 / USB-TTL 回环 / 虚拟串口对 / TCP loopback / UDP loopback |
| 串口名或地址 | COM3 / /dev/ttyUSB0 / 127.0.0.1:<port> |
| 串口配置 | baudrate=115200, data_bits=8, parity=none, stop_bits=1 |
| 占用检查 | 已关闭外部串口工具 / 不适用 |
| 权限检查 | 已确认 / 不适用 |

## 操作摘要

| 序号 | 工具 | request_id | handle_id | 结果 | 状态变化 | 备注 |
| --- | --- | --- | --- | --- | --- | --- |
| 1 | instance_create | req_xxx | h_ser_001 | ok | none -> Created | 创建 serial 实例 |
| 2 | serial_config | req_xxx | h_ser_001 | ok | Created -> Configured | 写入串口配置 |
| 3 | port_connect | req_xxx | h_ser_001 | ok | Configured -> Connected | 连接成功 |

## 收发帧摘要

| 序号 | 时间 | 方向 | request_id | encoding | bytes | preview | truncated | 备注 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| 1 | 12:00:01.000Z | tx | req_xxx | text | 6 | `ping\n` | false | 发送样例 |
| 2 | 12:00:01.020Z | rx | req_xxx | text | 6 | `ping\n` | false | 回环收到 |

## 错误与警告

| 时间 | request_id | error_id | category | code | retryable | recovery_hint | 处理结果 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| - | - | - | - | - | - | - | - |

## 日志摘要

| 时间 | event | request_id | handle_id | code | message | sensitive |
| --- | --- | --- | --- | --- | --- | --- |
| 12:00:01.000Z | tool_call | req_xxx | h_ser_001 | - | port_connect succeeded | false |

## 验收清单

- [ ] 目标资源可枚举或可创建。
- [ ] 连接成功后状态进入 Connected。
- [ ] text 发送和读取路径通过。
- [ ] hex 发送和读取路径通过。
- [ ] 资源独占冲突返回结构化错误。
- [ ] 断开和释放后资源可重新使用。
- [ ] 工具返回和日志均未泄漏敏感信息。

## 结论

- 结果：通过 / 不通过 / 未完成 / 需复测
- 阻塞问题：无 / <列表>
- 已知限制：无 / <列表>
- 后续动作：无 / <列表>
```

## JSONL 事件流格式

JSONL 文件每行一个 JSON 对象。事件流不替代结构化日志，但可作为调试报告的可分析附件。

### 会话开始事件

```json
{
  "type": "session_start",
  "timestamp": "2026-05-25T12:00:00.000Z",
  "record_id": "debug_20260525_windows_serial_com3_loopback",
  "platform": "windows",
  "port_mcp_version": "commit-or-build-id",
  "mcp_client": {
    "name": "client-name",
    "version": "client-version"
  },
  "scenario": "serial_loopback_acceptance",
  "resource": {
    "kind": "serial",
    "display": "COM3"
  },
  "sensitive": false
}
```

### 工具调用事件

```json
{
  "type": "tool_call",
  "timestamp": "2026-05-25T12:00:01.000Z",
  "request_id": "req_20260525_000001",
  "tool": "port_connect",
  "handle_id": "h_ser_001",
  "state_before": "Configured",
  "state_after": "Connected",
  "ok": true,
  "duration_ms": 18,
  "warnings": [],
  "sensitive": false
}
```

工具调用事件只保存参数摘要，不保存完整 payload。失败时应附带脱敏后的错误字段：

```json
{
  "type": "tool_call",
  "timestamp": "2026-05-25T12:00:03.000Z",
  "request_id": "req_20260525_000002",
  "tool": "port_connect",
  "handle_id": "h_ser_002",
  "state_before": "Configured",
  "state_after": "Error",
  "ok": false,
  "duration_ms": 9,
  "error": {
    "error_id": "err_20260525_000007",
    "category": "ResourceBusy",
    "code": "SERIAL_PORT_BUSY",
    "retryable": false,
    "message": "Serial port COM3 is already owned by another connected instance.",
    "recovery_hint": "Disconnect the owning instance or choose another serial port."
  },
  "sensitive": false
}
```

### 收发帧事件

```json
{
  "type": "frame",
  "timestamp": "2026-05-25T12:00:04.000Z",
  "handle_id": "h_ser_001",
  "request_id": "req_20260525_000003",
  "direction": "tx",
  "encoding": "text",
  "byte_len": 6,
  "preview": "ping\\n",
  "preview_encoding": "utf8",
  "truncated": false,
  "omitted_bytes": 0,
  "sensitive": false
}
```

`direction` 只允许 `tx` 或 `rx`。`preview` 必须按配置裁剪，不能包含完整大 payload。

### 日志摘要事件

```json
{
  "type": "log_summary",
  "timestamp": "2026-05-25T12:00:05.000Z",
  "event": "task_error",
  "request_id": null,
  "handle_id": "h_ser_001",
  "category": "WriteFailed",
  "code": "READ_IO_FAILED",
  "message": "Receive task exited with an I/O error.",
  "cause_chain_summary": [
    { "kind": "io", "message": "permission_denied" }
  ],
  "sensitive": false
}
```

### 会话结束事件

```json
{
  "type": "session_end",
  "timestamp": "2026-05-25T12:05:00.000Z",
  "record_id": "debug_20260525_windows_serial_com3_loopback",
  "result": "passed",
  "blocking_issues": [],
  "known_limits": [],
  "sensitive": false
}
```

## 字段约定

| 字段 | 说明 |
| --- | --- |
| `record_id` | 单次调试记录 ID，建议由日期、平台、资源和场景组成。 |
| `request_id` | 工具调用 ID，必须能关联工具返回和日志。 |
| `handle_id` | 实例句柄；释放后仍可作为历史记录引用。 |
| `error_id` | `last_error` 或错误事件 ID。 |
| `timestamp` | UTC 时间，格式使用 RFC3339。 |
| `resource.display` | 可读资源摘要，例如 `COM3` 或 `127.0.0.1:<port>`。 |
| `preview` | 裁剪后的 payload 预览。 |
| `omitted_bytes` | 因裁剪省略的字节数。 |
| `sensitive` | 当前记录是否含敏感字段或敏感摘要。 |

## 收发帧记录规则

| 规则 | 说明 |
| --- | --- |
| 默认只存摘要 | 保存方向、长度、编码、preview、截断状态和请求关联。 |
| preview 必须裁剪 | 默认不超过订阅 payload 上限或更小的调试预览上限。 |
| 二进制优先 hex preview | 非 UTF-8 payload 使用 hex preview，并记录 `preview_encoding=hex`。 |
| 不保存完整敏感数据 | 设备返回的序列号、认证响应、客户数据等必须脱敏或省略。 |
| 发送失败也可记录 | 如果 payload 已通过参数校验但底层写入失败，可记录 tx 摘要和错误事件。 |
| rx 溢出要记录统计 | 普通 `rx` 丢旧保新时记录 `rx_dropped_bytes` 或对应警告摘要。 |

## 错误链与恢复过程记录

错误记录应同时服务问题复盘和 AI 后续恢复。推荐至少保存：

| 字段 | 来源 |
| --- | --- |
| `request_id` | 工具返回。 |
| `error_id` | `last_error` 或错误转换层。 |
| `category` / `code` | [07-错误模型与返回格式](07-错误模型与返回格式.md)。 |
| `state_before` / `state_after` | 工具调用或日志事件。 |
| `resource.display` | 脱敏资源摘要。 |
| `retryable` | 工具失败返回。 |
| `recovery_hint` | 工具失败返回。 |
| `next_action_taken` | 人工或 AI 实际采取的下一步。 |
| `resolved` | 是否已恢复。 |

恢复过程示例：

```markdown
| 序号 | 错误 | recovery_hint | 实际动作 | 结果 |
| --- | --- | --- | --- | --- |
| 1 | ResourceBusy/SERIAL_PORT_BUSY | Disconnect the owning instance or choose another serial port. | 关闭外部串口工具后重试 port_connect | 已恢复 |
```

## 手工验收记录模板

Windows 串口手工验收建议使用以下补充表：

```markdown
## Windows 串口手工验收

| 项目 | 结果 | 证据摘要 |
| --- | --- | --- |
| 设备可枚举 | 通过 / 不通过 | port_scan 返回 COM3 |
| 连接成功 | 通过 / 不通过 | port_connect 后 Connected |
| text 回环 | 通过 / 不通过 | 发送 `ping\n`，收到同样字节 |
| hex 回环 | 通过 / 不通过 | 发送 `010203`，收到同样字节 |
| 独占冲突 | 通过 / 不通过 | 第二连接返回 SERIAL_PORT_BUSY |
| 错误路径 | 通过 / 不通过 | 不存在串口返回结构化错误 |
| 断开释放 | 通过 / 不通过 | release 后可重新连接 |
| 脱敏检查 | 通过 / 不通过 | 返回与日志无用户名、完整路径和堆栈 |

结论：通过 / 不通过 / 需复测
```

TCP/UDP loopback 验收可使用同样结构，替换为连接、回显、地址冲突、超时和释放重用等项目。

## 附件索引格式

如果需要保存截图、外部工具日志或设备资料，调试报告中只保存索引和脱敏摘要：

```markdown
## 附件索引

| 编号 | 类型 | 路径或引用 | 脱敏状态 | 说明 |
| --- | --- | --- | --- | --- |
| A1 | screenshot | attachments/device-manager-com3.png | 已脱敏 | Windows 设备管理器端口截图 |
| A2 | external-log | attachments/serial-tool-loopback.log | 已裁剪 | 外部串口工具回环日志摘要 |
```

附件不得默认提交完整原始日志。含客户数据、认证信息、环境变量或完整路径的附件必须先脱敏。

## 脱敏与保留策略

| 信息类型 | 记录策略 |
| --- | --- |
| 串口名 | 可记录，例如 `COM3` 或 `/dev/ttyUSB0`。 |
| IP 和端口 | 可记录本地或测试地址；生产地址按项目策略脱敏。 |
| 系统用户名 | 不记录。 |
| 完整本地路径 | 默认不记录，只保留文件名或路径类别。 |
| 环境变量值 | 不记录。 |
| 原始 payload | 默认不记录完整值，只存 preview 和长度。 |
| OS 错误文本 | 只保存脱敏摘要和 `io_kind`。 |
| 堆栈 | 默认不进入调试报告；如内部分析需要，必须单独标记 sensitive。 |

调试记录建议按用途分级保存：

| 级别 | 保存内容 | 用途 |
| --- | --- | --- |
| 可提交摘要 | Markdown 报告、脱敏表格、结论。 | 仓库内长期追踪。 |
| 本地附件 | 截图、外部工具日志、已按本文规则完成脱敏且可共享的 JSONL。 | 个人或团队短期排查。 |
| 敏感原始材料 | 未脱敏设备输出、客户环境日志、仅标记 `sensitive` 或包含客户/认证/环境信息的原始日志和 JSONL。 | 默认不提交、不进入 issue/PR；最长保留 14 天，仅排查责任人可访问，到期删除。 |

敏感原始材料提交或共享前必须先通过脱敏检查清单；无法脱敏的材料只能保留在受控本地位置，并记录责任人、创建时间、计划删除时间和删除确认。
