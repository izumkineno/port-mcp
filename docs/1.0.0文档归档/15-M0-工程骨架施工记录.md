## M0 目标

M0 的目标是建立可持续编译、可继续扩展、边界清晰的 Rust 工程骨架，并用最小代码证明官方 Rust MCP SDK 能支撑初版 stdio server 接入。

M0 完成后应具备：

- `Cargo.toml` 写入经复核的基础依赖和最小 feature。
- `Cargo.lock` 生成或更新，构建可复现。
- `src/` 下建立 `mcp`、`app`、`runtime`、`transport`、`model`、`util` 模块目录或模块文件。
- `main.rs` 从占位输出变为薄入口，只负责配置、日志、运行时和 MCP server 启动装配。
- `mcp` 层完成最小 `rmcp` stdio server spike。
- 至少一个临时 smoke tool 能证明工具注册、参数解析和返回包装可用；该工具不得进入最终初版工具列表。
- 能观察并记录 `RequestContext<RoleServer>` 或等价上下文的 session 能力结论。
- 能构造或发送最小 resource notification 证明订阅通知路径可对接。
- `cargo check` 通过，并记录命令、日期、平台和关键输出。

## M0 不做事项

M0 只搭骨架和复核 SDK，不实现业务能力。

| 不做事项 | 原因 |
| --- | --- |
| 不实现真实工具逻辑 | 工具契约由 03 固化，真实服务应从 M1/M2 后开始接入。 |
| 不打开串口、TCP socket 或 UDP socket | 真实 I/O 属于 M5/M6，M0 只锁定依赖和模块边界。 |
| 不实现 RuntimeRegistry | RuntimeRegistry 属于 M2，M0 只预留模块入口。 |
| 不实现运行时状态机、资源锁、队列或缓冲区 | M1 只定义状态模型、状态摘要和返回模型；运行时状态机、生命周期迁移和资源锁属于 M2；队列和缓冲区属于 M4。 |
| 不把 MCP SDK 类型穿透到 `app`、`runtime`、`transport` 或 `model` | 09、13、14 已要求 SDK 类型只能封装在 `mcp` 层。 |
| 不引入 HTTP transport、OAuth、VISA、协议 helper、数据库或 Web 框架 | 都不属于初版 M0 依赖边界。 |

## 施工步骤

### 1. 更新依赖基线

按 14 的推荐入口加入最小依赖，并在实际编译后记录版本和 feature 差异。

建议起点：

```toml
[dependencies]
rmcp = { version = "1.7.0", features = ["server", "macros", "schemars", "transport-io"] }
tokio = { version = "1", features = ["macros", "rt-multi-thread", "sync", "time", "net", "io-util"] }
serde = { version = "1", features = ["derive"] }
serde_json = "1"
thiserror = "2"
tracing = "0.1"
tracing-subscriber = { version = "0.3", features = ["fmt", "env-filter"] }
bytes = "1"
serialport = "4"
time = { version = "0.3", features = ["formatting", "parsing", "macros"] }
```

如果本地 registry 仍因 `rsproxy-sparse` 无法查询 crates.io，应记录实际命令、错误和替代方式，例如显式 `--registry crates-io` 或确认镜像源可用。

证据回填：

| 检查项 | 结果 | 证据 |
| --- | --- | --- |
| `rmcp` 版本与 feature 已确认 | 通过 | `Cargo.toml` 使用 `rmcp = "1.7.0"`，runtime feature 为 `server/macros/schemars/transport-io`；dev-dependency 仅为 M0 内存 smoke 测试启用 `client`。 |
| `tokio` feature 最小化 | 通过 | `tokio` feature 为 `macros/rt-multi-thread/sync/time/net/io-util`，支撑 async main、duplex smoke 与后续 transport 基线。 |
| `serialport` 未用于真实 I/O | 通过 | `serialport = "4"` 已锁定但 `src/transport` 仍为空模块，未打开串口。 |
| 未引入 M0 外依赖 | 通过 | 依赖限于 14 推荐基线；未加入 HTTP/OAuth/VISA/数据库/Web 框架。 |
| `Cargo.lock` 已生成或更新 | 通过 | `cargo check` 已成功完成并更新 `Cargo.lock`。 |

### 2. 建立模块骨架

按 13 推荐边界建立模块：

```text
src/
  main.rs
  mcp/
  app/
  runtime/
  transport/
  model/
  util/
```

M0 模块只需要提供可编译的最小入口，不应填入跨阶段业务逻辑。

建议最小职责：

| 模块 | M0 最小职责 | 不得提前做 |
| --- | --- | --- |
| `mcp` | 封装 `rmcp` stdio server、临时 smoke tool、上下文探测和通知 spike。 | 不直接管理实例状态或底层 I/O。 |
| `app` | 预留应用服务入口和 smoke handler 的边界适配。 | 不实现初版工具业务。 |
| `runtime` | 预留 RuntimeRegistry 类型或构造入口占位。 | 不实现实例表、资源锁或后台任务。 |
| `transport` | 预留 transport trait 或模块边界。 | 不打开串口、TCP 或 UDP。 |
| `model` | 预留跨层模型模块。 | 不一次性铺完 M1 模型。 |
| `util` | 预留日志、时间、ID、限制等工具模块。 | 不复制 06/07/11 的完整逻辑。 |

证据回填：

| 检查项 | 结果 | 证据 |
| --- | --- | --- |
| 顶层模块可编译 | 通过 | `cargo check` 通过；`src/main.rs` 挂载 `mcp/app/runtime/transport/model/util`。 |
| 依赖方向符合 13 | 通过 | 入口只调用 `mcp::server::run_stdio_server()`；`app/runtime/transport/model/util` 未反向依赖 `mcp`。 |
| `rmcp` 类型只出现在 `mcp` 层 | 通过 | `Select-String -Path src\*.rs,src\*\*.rs -Pattern 'rmcp'` 仅命中 `src/main.rs` 的返回类型 glue 与 `src/mcp/*`。 |
| 业务逻辑未提前进入 M0 | 通过 | 仅实现 `m0_smoke` 临时 SDK smoke tool，未实现 03 的真实端口工具。 |

### 3. 替换薄 `main.rs`

`main.rs` 应从占位输出改为薄入口。M0 不要求完整配置系统，但必须避免继续输出 `Hello, world!` 作为服务行为。

M0 最小入口应完成：

- 初始化 `tracing` 或等价日志 subscriber。
- 读取或构造默认运行配置摘要。
- 创建运行时骨架或传入后续 RuntimeRegistry 构造占位。
- 调用 `mcp::server` 的 stdio server 启动入口。
- 启动失败时返回可诊断错误，不把常规启动失败写成裸 `panic`。

证据回填：

| 检查项 | 结果 | 证据 |
| --- | --- | --- |
| `main.rs` 不再输出占位文本 | 通过 | `Hello, world!` 已删除；stdout 不写启动文本，避免污染 MCP stdio。 |
| 启动日志包含版本或配置摘要 | 通过 | 启动日志包含 `CARGO_PKG_NAME`、`CARGO_PKG_VERSION`、`session_mode=unverified_m0_spike`，writer 为 stderr。 |
| 启动路径保持薄入口 | 通过 | `main.rs` 只初始化 tracing 并调用 `mcp::server::run_stdio_server()`。 |
| 常规失败路径可诊断 | 通过 | `run_stdio_server()` 对 serve 初始化错误写 `tracing::error!` 并返回 `rmcp::RmcpError`。 |

### 4. 完成 SDK spike

SDK spike 用来把 14 的复核结论变成本仓库可编译证据。临时 smoke tool 可以存在于 M0 spike 中，但必须标注为临时，不进入 03 的初版工具契约。

必须证明：

| 能力 | M0 证明方式 | 结果 |
| --- | --- | --- |
| stdio server | `mcp::server` 能创建并运行 `rmcp` stdio server。 | 通过：`M0SmokeServer::new().serve(stdio()).await?.waiting().await?` 可编译。 |
| tool 注册 | 临时 smoke tool 可被 SDK 注册。 | 通过：`cargo test` 通过 `m0_smoke` tool call。 |
| typed params/schema | smoke tool 参数可由 `serde`/`schemars` 或 SDK schema 机制表达。 | 通过：`SmokeParams` 使用 `serde::Deserialize` 与 `schemars::JsonSchema`，并通过 `Parameters<SmokeParams>` 解析。 |
| request context | handler 能获得 `RequestContext<RoleServer>` 或等价上下文。 | 通过：`m0_smoke` handler 直接接收 `RequestContext<RoleServer>` 并记录 request id/peer info。 |
| stable session id | 明确记录可用、不可用或只能连接级区分。 | 不采用：M0 stdio request context 可提供 request id、peer、meta、extensions，但未提供稳定 AI session id；M1 默认按 `session_mode=unavailable` 处理。 |
| resource notification | 能构造或发送最小 resource updated/list changed notification。 | 通过：`m0_smoke` 发送 `notify_resource_updated(ResourceUpdatedNotificationParam::new("port-mcp://m0/smoke"))`，测试客户端收到通知。 |
| error response | SDK 错误可在 `mcp` 层转换为本项目统一错误外壳。 | 部分通过：M0 将 notification send error 转为 `McpError::internal_error`；项目统一错误模型仍按 M1/M7 设计落地。 |

session 结论必须回填到下表：

| 结果 | 是否采用 | 后续影响 |
| --- | --- | --- |
| `session_mode=auto` | 不采用 | M0 未发现 stdio request context 中可直接作为稳定 AI session id 的字段。 |
| `session_mode=unavailable` | 采用 | M1 必须禁用依赖会话身份的默认句柄与订阅隔离路径；仍可使用显式 handle 参数。 |
| `session_mode=single_dev` | 暂不采用 | 仅作为未来本地显式开发模式备选，不能作为默认行为。 |

- `Cargo.toml` 已加入 M0/M1 必需依赖，feature 选择有记录。

- `src/` 模块骨架已建立，并能通过 `cargo check`。
- `main.rs` 已变成薄入口，不再保留占位输出。
- `mcp` 层完成最小 `rmcp` stdio server spike。
- session 能力结论已记录，并明确 M1 的 `session_mode` 路径。
- resource notification 能力已通过最小 spike 或记录了退化方案。
- `rmcp` 类型没有穿透到 `app`、`runtime`、`transport`、`model`。
- 未提前实现 M1/M2/M3 的业务逻辑。
- 本文档的证据回填表已更新，不再只有“待填”。

## 回写规则

出现以下情况时，应先更新本文和相关上游文档，再继续施工：

- M0 实际依赖与 14 推荐入口不一致。
- `rmcp` feature 组合无法通过 `cargo check`。
- 本地 registry 或镜像源导致依赖解析策略变化。
- `main.rs` 为了启动 SDK 必须承担超出薄入口的职责。
- SDK 无法提供 request context、resource notification 或稳定错误映射。
- session 能力结论影响 03、04、05、06 中的默认句柄或订阅隔离设计。
