# 04 端口 MCP 运行时架构设计

本文档承接 [01-初版工程功能设计大纲](01-初版工程功能设计大纲.md) 与 [03-初版工具契约](03-初版工具契约.md)，明确初版工程的运行时技术选型、目录结构、模块职责、全局实例表、后台任务模型和 MCP server 接入方式。

本文档只聚焦初版运行时架构。状态转移细节、缓冲区容量策略、完整错误码、测试矩阵和运行配置分别由后续 `05` 到 `10` 文档承接。

## 架构结论

- 初版采用 Rust 原生 MCP server 入口，不再以 TypeScript 作为协议壳。
- 异步运行时采用 `tokio`，所有端口 I/O、后台任务、队列和取消信号统一运行在同一个 Tokio runtime 中。
- MCP 工具层只负责参数解析、会话上下文识别、调用应用服务和包装返回，不直接操作串口或 socket。
- 运行时核心采用轻量混合 Transport 架构：收发任务、队列、统计、订阅广播使用统一模型；串口、TCP、UDP 的配置校验、连接建立和资源占用按类型分支实现。
- `MockTransport` 作为一等模块纳入初版架构，用于早期开发、单元测试和无硬件环境下验证工具契约。

## 运行时分层

初版运行时按从外到内分为四层：

| 层级 | 职责 | 不负责 |
| --- | --- | --- |
| MCP 接入层 | 启动 server、注册工具、解析入参、识别 AI 会话、返回 MCP 结果 | 直接读写端口、保存底层资源句柄 |
| 应用服务层 | 实现工具用例、校验状态前置条件、协调实例表和资源锁 | 持有具体串口或 socket 的读写循环 |
| 运行时核心层 | 管理实例、任务句柄、发送队列、接收缓冲区、订阅广播和统计 | 理解 MCP 协议细节 |
| Transport 层 | 打开、关闭、读取、写入具体底层资源 | 决定工具返回格式和 AI 会话绑定 |

推荐调用路径如下：

```text
MCP Client
-> Rust MCP Server
-> Tool Handler
-> PortService / InstanceService
-> RuntimeRegistry
-> RuntimeInstance
-> TransportTask
-> Serial/TCP/UDP/Mock Transport
```

### 模块职责

| 模块 | 职责 |
| --- | --- |
| `mcp::server` | 创建 Rust 原生 MCP server，绑定 stdio 或后续可选传输方式，注册所有初版工具。 |
| `mcp::tools` | 将 MCP 工具请求映射到应用服务方法，保持工具名与 [03-初版工具契约](03-初版工具契约.md) 一致。 |
| `mcp::session` | 提取当前 AI 会话标识，维护会话默认实例绑定。 |
| `mcp::response` | 将应用层成功结果和错误转换为统一 MCP 返回对象。 |
| `app::*_service` | 承载工具用例逻辑，例如创建实例、配置实例、连接、发送、读取、订阅。 |
| `runtime::registry` | 保存全局实例表、会话绑定表和共享资源锁入口。 |
| `runtime::instance` | 表示单个连接实例的内存状态、配置、统计、缓冲区、队列和任务句柄。 |
| `runtime::tasks` | 定义连接后的接收任务、发送任务、任务取消和任务退出上报。 |
| `runtime::queues` | 定义每实例发送队列和内部控制消息。 |
| `runtime::buffers` | 定义普通 `rx` 缓冲区接口，容量和溢出细节由 `06` 文档细化。 |
| `runtime::subscription` | 管理每实例订阅者列表和广播入口。 |
| `runtime::resource_locks` | 管理串口独占锁、TCP listen 地址锁、UDP bind 地址锁。 |
| `transport::*` | 实现串口、TCP、UDP、Mock 的打开、读、写、关闭能力。 |
| `model::*` | 定义连接配置、实例状态、错误类型、编码数据等跨层模型。 |
| `util::*` | 放置时间、编码、限制常量等无状态工具。 |

## 核心数据模型

### 全局运行时

运行时入口建议为 `RuntimeRegistry`，由 MCP server 启动时创建并注入所有工具 handler。

```text
RuntimeRegistry
- instances: handle_id -> RuntimeInstance
- session_bindings: session_id -> handle_id
- resource_locks: ResourceLockTable
- id_generator: HandleIdGenerator
- limits: RuntimeLimits
```

`RuntimeRegistry` 是跨工具共享状态的唯一入口。工具 handler 不应绕过它保存自己的实例缓存。

### 连接实例

单个实例建议由 `RuntimeInstance` 表示：

```text
RuntimeInstance
- handle_id
- instance_type: Serial | TCP | UDP | Mock
- state
- config
- created_at
- connected_at
- last_activity_at
- stats
- rx_buffer
- tx_queue
- subscriptions
- task_group
- last_error
```

`Mock` 不暴露为初版用户工具的连接类型，除非测试或开发模式显式启用。它作为内部测试 Transport 类型存在，用来验证实例表、队列、缓冲区、订阅和错误路径。

### 状态共享方式

初版建议使用 `Arc` 持有全局 registry，实例内部使用异步锁保护可变状态。锁粒度应按实例拆分，避免一个实例的长时间 I/O 阻塞其他实例。

建议原则：

- 全局实例表锁只保护插入、删除和查找实例引用。
- 单实例状态锁保护状态、配置、统计、缓冲区和任务句柄的短临界区修改。
- 底层 I/O 不应在持有实例状态锁时执行。
- 发送和接收任务通过 channel 与实例状态交互，避免工具调用直接抢占 I/O 资源。

## MCP Server 接入

### 启动流程

`main.rs` 的职责应保持很薄：

1. 加载运行时限制和基础配置。
2. 初始化日志或追踪设施。
3. 创建 `RuntimeRegistry`。
4. 创建 MCP server。
5. 注册 [03-初版工具契约](03-初版工具契约.md) 中定义的所有初版工具。
6. 通过 stdio 运行 MCP server。

### 工具 handler 规则

每个工具 handler 遵循同一流程：

```text
parse args
-> resolve session_id
-> call app service
-> map success/error to MCP response
```

handler 不做以下事情：

- 不直接打开串口、TCP socket 或 UDP socket。
- 不直接修改实例状态枚举。
- 不直接读写 `rx` 缓冲区或发送队列。
- 不吞掉应用层错误。

这样可以确保 MCP 协议变化时只影响 `mcp` 模块，端口运行时核心仍可独立测试。

## 应用服务边界

应用服务层建议按工具组拆分：

| 服务 | 覆盖工具 | 主要职责 |
| --- | --- | --- |
| `InstanceService` | `instance_create`、`instance_list`、`instance_query`、`instance_use`、`instance_release` | 管理实例生命周期入口和会话绑定。 |
| `ConfigService` | `serial_config`、`tcp_udp_config` | 校验配置并写入实例，必要时重置配置相关错误。 |
| `PortService` | `port_scan`、`port_connect`、`port_disconnect`、`port_send`、`port_pull`、`port_clear` | 管理连接、断开、发送、读取和缓冲区清理。 |
| `StreamService` | `port_subscribe_stream`、`port_unsubscribe_stream` | 管理当前 AI 会话对实例的订阅关系。 |

服务层需要保证：

- 所有状态前置条件在服务层检查。
- 所有资源锁申请和释放由服务层通过 `RuntimeRegistry` 统一协调。
- 所有返回数据使用 `model` 中的结构，不直接拼装松散 JSON。
- 所有错误统一转换为 `PortMcpError` 或同等应用错误类型。

## Transport 架构

### 轻量混合抽象

初版不要求把 Serial、TCP、UDP 的所有行为压成完全一致的接口。推荐拆为两段：

1. 类型分支负责配置校验、资源锁 key 生成和连接建立。
2. 连接成功后返回统一的读写句柄，交给通用发送/接收任务管理。

统一读写句柄至少需要支持：

```text
read_chunk(max_bytes, timeout_ms) -> bytes
write_all(bytes, timeout_ms) -> written_bytes
close() -> result
```

具体实现可因底层库而异，但进入 `runtime::tasks` 后应表现为同一套收发语义。

### Serial Transport

`transport::serial` 负责：

- 校验串口参数是否满足 [01-初版工程功能设计大纲](01-初版工程功能设计大纲.md) 的初版范围。
- 打开本机串口。
- 将串口读写适配为统一读写句柄。
- 在打开前申请串口独占锁，关闭后释放锁。

串口枚举由 `port_scan(type=Serial)` 使用同一模块提供的扫描能力。

### TCP Transport

`transport::tcp` 负责：

- `mode=client` 时主动连接远端地址。
- `mode=server/listen` 时监听本地地址，并按初版策略接收客户端连接。
- 为 listen 地址申请资源锁，避免同一地址端口重复监听。
- 将已建立连接适配为统一读写句柄。

TCP server/listen 初版固定采用最小单客户端策略：同一实例最多维护一个当前活动连接。第一个客户端成为 `active_peer`；后续客户端必须被拒绝或立即关闭，并记录可诊断事件。`instance_query` 必须可观察 `listen_strategy=single_client`、`active_peer` 和 `connected_client_count`。多客户端 listen 需要引入客户端子句柄或来源元数据，属于进阶能力，不进入初版。

### UDP Transport

`transport::udp` 负责：

- 根据 `mode=client` 或 `mode=server/listen` 创建 UDP socket。
- 管理远端地址绑定或最近远端地址策略。
- 为 bind/listen 地址申请资源锁。
- 将 datagram 收发适配到统一收发任务。

UDP 是报文边界协议。初版为了保持 `port_pull` 与普通 `rx` 缓冲区的字节流契约一致，普通 `rx` 缓冲区明确丢弃 datagram 边界，只保证按接收顺序追加 payload 字节；订阅通知可以按单次接收 chunk 标记 `transport_protocol=udp` 和 `datagram=true`。需要可恢复 datagram 边界、来源地址队列或多来源帧元数据时，应作为进阶能力扩展返回契约。

### MockTransport

`transport::mock` 是初版一等模块，目标是让运行时核心先于真实硬件稳定。

MockTransport 至少支持：

- 可配置读入数据脚本。
- 记录写入数据。
- 模拟读超时、写失败、连接中断和缓冲区压力。
- 与真实 Transport 一样接入发送任务、接收任务、缓冲区和订阅广播。

MockTransport 的使用范围：

- 单元测试默认使用。
- 集成测试可通过开发配置显式启用。
- 正式 MCP 工具默认不向用户暴露 `type=Mock`，避免污染初版契约。

## 后台任务模型

连接成功后，每个实例至少启动两个后台任务：

| 任务 | 输入 | 输出 | 退出条件 |
| --- | --- | --- | --- |
| 接收任务 | 底层 Transport 读句柄、取消信号 | `rx_buffer`、订阅广播、统计更新 | 断开、释放、读错误、取消信号 |
| 发送任务 | 实例发送队列、底层 Transport 写句柄、取消信号 | 底层写入、统计更新、错误上报 | 断开、释放、写错误、队列关闭 |

任务组由 `runtime::tasks::TaskGroup` 管理，保存任务句柄、取消令牌和退出状态。

### 接收任务

接收任务是底层数据进入系统的唯一入口：

```text
read from transport
-> append to rx_buffer
-> update rx stats and last_activity_at
-> broadcast to subscriptions
-> report errors to instance last_error
```

接收任务不得直接生成 MCP 返回。订阅通知的协议包装由 `mcp` 层或专门的通知适配器处理。

### 发送任务

发送任务是底层写入的唯一执行者：

```text
receive send command
-> write full frame to transport
-> update tx stats and last_activity_at
-> report success/failure to waiting caller when needed
```

同一实例的多个 `port_send` 请求必须按进入队列顺序串行写入。单帧写入不可被其他帧插入。

### 断开与释放

`port_disconnect` 和 `instance_release` 都必须通过任务组执行统一清理：

1. 将状态标记为 `Disconnecting`。
2. 发出取消信号并关闭发送队列入口。
3. 等待接收任务和发送任务退出，设置合理超时。
4. 关闭底层 Transport。
5. 释放资源锁。
6. 将状态设置为 `Disconnected` 或从实例表移除。

详细状态转移和异常清理顺序由 `05-状态机与资源生命周期.md` 固化。

## 全局实例表与资源锁

### 实例表

实例表保存所有未释放实例：

```text
handle_id -> RuntimeInstance
```

约束：

- `handle_id` 在实例可查询期间不得复用。
- `Released` 实例应从表中移除。
- `instance_list` 只读取实例摘要，不应长时间持有实例锁。
- `instance_query` 可读取完整状态快照，但不得改变状态。

### 会话绑定表

会话绑定表保存当前 AI 会话的默认实例：

```text
session_id -> handle_id
```

约束：

- `instance_use` 只修改当前 `session_id` 的绑定。
- 实例相关工具显式传入 `handle_id` 时直接使用该句柄；未传入且工具允许缺省时，必须通过当前 `session_id` 查找默认实例。
- 释放实例时，应清理所有指向该实例的会话绑定。
- 如果 MCP SDK 无法提供稳定会话 ID，初版进入 `session_mode=unavailable`：禁止所有依赖当前会话身份的路径，包括默认实例解析、`instance_use`、`port_subscribe_stream` 和 `port_unsubscribe_stream`，返回 `InvalidState/SESSION_ID_UNAVAILABLE`；所有实例相关工具必须显式传 `handle_id`。仅在用户显式启用单会话开发模式时，才允许使用进程级伪会话，并在工具返回和日志中标记 `session_mode=single`。`session_mode=single_dev` 仅允许显式本地开发配置启用，远程或多客户端入口必须拒绝，启用后必须拒绝第二客户端或等价多会话入口。

### 资源锁表

资源锁表负责阻止底层资源冲突：

| 资源类型 | 锁 key | 初版策略 |
| --- | --- | --- |
| 串口 | 规范化串口名，例如 `serial:COM3` | 独占 |
| TCP listen | `tcp-listen:ip:port` | 独占 |
| UDP bind/listen | `udp-bind:ip:port` | 默认独占 |

TCP client 不默认加远端地址锁，因为多个 TCP client 连接同一远端是常见合法场景。

## 数据流

### 创建与配置

```text
instance_create
-> RuntimeRegistry alloc handle_id
-> insert RuntimeInstance(state=Created)

serial_config / tcp_udp_config
-> validate type and args
-> write config
-> state = Configured
```

配置工具不得打开底层资源。

### 连接

```text
port_connect
-> check state Configured | Disconnected
-> acquire resource lock
-> open concrete transport
-> create queues and task group
-> start receive/send tasks
-> state = Connected
```

如果打开 Transport 或启动任务失败，必须释放已申请的资源锁，并将错误写入 `last_error`。

### 发送

```text
port_send
-> validate Connected
-> decode text/hex payload
-> enqueue send command
-> wait for queue acceptance or write result according to implementation choice
-> return sent_bytes / queued
```

初版工具契约允许返回 `queued=true`。如果实现选择等待实际写入完成，必须仍保持队列串行语义。

### 拉取读取

```text
port_pull
-> validate Connected
-> wait until rx_buffer has data or timeout
-> consume up to max_bytes
-> return data with encoding metadata
```

`port_pull` 只消费普通 `rx` 缓冲区，不影响订阅者已经收到或将收到的广播事件。

### 订阅

```text
port_subscribe_stream
-> validate handle and session_id
-> register subscriber for handle_id + session_id
-> return stream uri

receive task
-> broadcast received chunk to all subscribers

port_unsubscribe_stream
-> remove current session subscriber
```

订阅列表属于实例状态，但每个订阅者以 `session_id` 区分，避免一个 AI 会话取消订阅影响其他会话。

## 依赖建议

具体 crate 可在实现阶段按维护状态确认，但初版架构按以下能力预留：

| 能力 | 建议方向 |
| --- | --- |
| 异步运行时 | `tokio` |
| 序列化 | `serde`、`serde_json` |
| 错误定义 | `thiserror` 或等价方案 |
| 日志追踪 | `tracing`、`tracing-subscriber` |
| 串口 | 支持 Windows/Linux/macOS 的 Rust serial crate |
| TCP/UDP | `tokio::net` |
| 取消信号 | `tokio-util` 的 cancellation token 或等价模式 |

MCP server SDK 的具体 crate 名称应在实现前再次确认官方文档和维护状态。无论选用哪个 SDK，本文档定义的分层边界不变。

## 平台边界

初版必须把 Windows 作为主要开发与验证平台之一，因为串口名、权限和换行行为会影响调试体验。

跨平台约束：

- 串口资源名需要保留原始显示名，同时内部锁 key 应规范化。
- 网络监听地址应明确 IPv4、IPv6 和 `0.0.0.0` 的表现。
- 错误信息不得泄露无关用户目录、环境变量或凭据。
- 串口依赖缺失或权限不足时，应返回结构化错误，而不是 panic。

## 初版实现顺序建议

1. 建立 `model`、`runtime::registry`、`RuntimeInstance` 和 `HandleIdGenerator`。
2. 接入 Rust 原生 MCP server，注册空实现工具，统一成功和失败返回结构。
3. 实现 `InstanceService` 与实例管理工具。
