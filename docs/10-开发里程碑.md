# 10 端口 MCP 开发里程碑

本文档承接 [03-初版工具契约](03-初版工具契约.md)、[04-运行时架构设计](04-运行时架构设计.md)、[05-状态机与资源生命周期](05-状态机与资源生命周期.md)、[06-并发与缓冲区设计](06-并发与缓冲区设计.md)、[07-错误模型与返回格式](07-错误模型与返回格式.md)、[08-测试与验收计划](08-测试与验收计划.md) 与 [09-库选择与依赖决策](09-库选择与依赖决策.md)，把初版工程拆成可连续提交、可独立验证、可回滚的实现阶段。

本文档不替代具体任务管理，也不要求每个里程碑只能对应一个 commit。它定义的是工程推进顺序：每个阶段完成后，都应能通过明确的自动化检查或手工验收进入下一阶段。

## 设计结论

- 初版实现按可提交阶段推进，而不是按文件目录一次性铺开。
- 每个阶段必须有清晰产物、验收方式和进入下一阶段的门槛。
- 先建立模型、错误外形和 MockTransport，再接入真实 TCP/UDP，最后接入串口和 MCP server。
- 自动化验证优先覆盖无硬件路径：单元测试、MockTransport 集成测试、TCP/UDP loopback、并发和错误模型。
- 串口使用 `serialport`，但不进入 CI 硬件门槛；真实串口能力在 Windows 手工验收阶段确认。
- 任一阶段发现 03 到 09 的设计冲突，应先回写对应文档，再继续实现。

## 里程碑原则

| 原则 | 说明 |
| --- | --- |
| 可提交 | 阶段结束时仓库应处于可构建、可测试、可解释的状态。 |
| 可验证 | 每个阶段至少有一种明确验证方式，不能只靠代码阅读判断完成。 |
| 低耦合 | MCP 协议层、应用服务层、运行时核心层和 Transport 层按 04 的边界逐步连通。 |
| 无硬件优先 | CI 先覆盖 MockTransport 与 loopback，避免早期被串口硬件阻塞。 |
| 错误先收敛 | 在真实 I/O 扩大前，先固化统一返回、错误码和状态影响。 |
| 只做初版 | VISA、协议 helper、心跳、重连和高级订阅过滤不进入初版里程碑。 |

## M0 工程骨架与依赖基线

### 目标

建立可以持续编译的 Rust 工程骨架，固定 09 中的初版依赖方向，并让后续模块有稳定落点。

### 实现范围

- 整理 `Cargo.toml`，加入 `tokio`、`serde`、`serde_json`、`thiserror`、`tracing`、`tracing-subscriber`、`bytes` 和 `serialport` 等初版依赖。
- 复核官方 Rust MCP SDK 的 crate、版本、许可证、stdio server、tool schema、request context、resource notification、error response 和 session/context 能力，提交最小 spike 或复核记录。
- 依赖 feature 只开启当前阶段或近期阶段需要的最小集合。
- 建立 04 推荐的模块目录：`mcp`、`app`、`runtime`、`transport`、`model`、`util`。
- 保持 `main.rs` 很薄，只初始化日志、运行时和后续 server 入口占位。
- 添加最小构建检查，确保空模块可以通过编译。

### 不做

- 不实现任何真实工具逻辑。
- 不打开串口、TCP socket 或 UDP socket。
- 不引入 02 中的进阶硬件或协议依赖。

### 验收门槛

- `cargo check` 通过。
- `Cargo.lock` 可生成并随工程保持可复现。
- 文档中列出的默认依赖没有出现不维护或与初版无关的替代库。
- 官方 Rust MCP SDK 的能力结论已记录；若没有稳定 session_id，M1 必须实现 `session_mode=unavailable` 退化模型。
- 若配置启用 `session_mode=single_dev` 开发模式，必须只允许显式本地配置启用，并在启动日志、工具返回和诊断中标记 `session_mode=single`。

SDK spike 结论必须按下表处理：

| 能力缺口 | M0/M1 gate |
| --- | --- |
| 无可用官方 Rust MCP SDK、许可证不兼容、无 stdio server、无工具注册或无稳定错误响应 | 停止进入 M2，先回写 09 并重新决策。 |
| 无稳定 session/context | 可继续，但 M1 必须实现 `session_mode=unavailable`，并禁用依赖当前会话身份的工具路径。 |
| 无 resource notification 能力 | 可继续，但必须在 M3/M4 用事件队列模拟订阅语义，并通过 08 的订阅广播测试。 |
| tool schema 能力不完整 | 可继续，但 `mcp` 层必须提供本项目 schema/参数校验封装，不能让松散参数进入 app service。 |

## M1 模型、返回与错误基线

### 目标

先固化跨层共享模型，让后续工具、运行时和 transport 都依赖同一套状态、配置、返回和错误结构。

### 实现范围

- 定义实例类型、实例状态、连接配置、统计字段和限制值模型。
- 实现统一成功返回和失败返回结构，覆盖 `ok`、`tool`、`request_id`、`timestamp`、`handle_id`、`state`、`data`、`warnings` 和 `error`。
- 定义 `PortMcpError` 或等价领域错误类型，覆盖 07 的 category 和主要 code。
- 实现错误到返回对象的转换层，默认执行脱敏和 details 裁剪。
- 实现进程内 `handle_id`、`request_id`、`error_id` 生成器。

### 不做

- 不实现实例表和工具服务。
- 不要求所有错误码一次性填满业务上下文，但 category、code 和返回外形必须稳定。

### 验收门槛

- 序列化单元测试覆盖成功返回、失败返回、警告返回和 `last_error` 摘要。
- 错误类别与 03、07 保持一致。
- 参数错误、状态错误、资源冲突和 I/O 错误能映射到稳定 category/code。

## M2 RuntimeRegistry 与实例生命周期

### 目标

实现无 I/O 的运行时核心，让实例生命周期、状态矩阵、会话绑定和资源锁可以先独立验证。

### 实现范围

- 实现 `RuntimeRegistry`、`RuntimeInstance`、`ResourceLockTable` 和 `RuntimeLimits`。
- `ResourceLockTable` 必须支持 `held` 与 `closing` tombstone 状态，避免强制释放后台 close 期间同资源被复用。
- 实现实例管理服务：`instance_create`、`instance_list`、`instance_query`、`instance_use`、`instance_release`。
- 实现配置服务：`serial_config`、`tcp_udp_config` 的参数校验和配置快照写入。
- 实现状态前置检查、状态提交和配置覆盖的原子性规则。
- 实现串口、TCP listen、UDP bind 的资源锁 key 规范化和冲突判断。

### 不做

- 不打开真实 transport。
- 不实现发送队列、接收任务或订阅广播。

### 验收门槛

- 05 的工具允许状态矩阵可通过单元测试验证。
- 资源锁测试覆盖申请成功、冲突失败、释放、强制释放进入 `closing` tombstone、后台 close 成功释放和 tombstone 超时诊断。
- `Released` 实例从实例表移除，同句柄后续返回 `HandleNotFound`。

## M3 MockTransport 与任务骨架

### 目标

用 MockTransport 打通 connect、send task、receive task、disconnect、release 和 Error 收敛路径，避免真实 I/O 过早干扰状态机。

### 实现范围

- 实现内部 `MockTransport`，仅用于测试和开发模式，不暴露为用户工具类型。
- 定义 transport 打开后返回的统一读写句柄或等价封装。
- 实现 task group、取消信号、任务退出原因和 sibling task 停止规则。
- 实现 `port_connect`、`port_disconnect` 的 Mock 路径。
- 实现后台任务致命错误写入 `last_error` 并进入 `Error`。

### 不做

- 不接入 TCP/UDP 或串口。
- 不完善订阅广播细节，只保留任务可插拔位置。

### 验收门槛

- MockTransport 基础生命周期测试通过：create -> config -> connect -> send -> pull -> disconnect -> release。
- connect 失败能释放资源锁并进入 `Error`。
- `instance_release(force=true)` 句柄立即失效，资源锁进入 `closing` tombstone，后台 close 成功或超时后才允许同资源复用。

## M4 队列、缓冲区与订阅

### 目标

补齐 06 定义的内存并发语义，让收发、拉取、清理和订阅在 MockTransport 下先稳定。

### 实现范围

- 实现每实例有界 `tx_queue`，满时拒绝新发送。
- 实现普通 `rx_buffer`，满时丢旧保新并累计 `rx_dropped_bytes`。
- 实现 `port_send`、`port_pull`、`port_clear`。
- 实现 `port_subscribe_stream`、`port_unsubscribe_stream` 和每订阅者独立队列。
- 实现统计字段：发送、接收、丢弃、订阅者数量、队列长度和最近活动时间。
- 实现多会话默认实例绑定与订阅隔离。
- 实现全局动态预算计数器：`buffer_bytes_budget`、`queued_bytes_budget`，以及 `ABS_MAX_TOTAL_BUFFER_BYTES`、`ABS_MAX_TOTAL_QUEUED_BYTES` 的配置校验。
- 实现最小通知频率保护：单订阅者、单实例、全局通知速率限制，burst、合并/丢弃策略和 `dropped_notifications` 统计。

### 不做

- 不引入跨会话公平调度。
- 不实现高级订阅过滤或协议解析。

### 验收门槛

- FIFO、tx 满拒绝、rx 丢旧保新、pull 消耗式读取、clear 语义测试通过。
- 慢订阅者只丢自己的通知，不阻塞接收任务或其他订阅者。
- 动态预算 reserve 失败、丢弃后重试、tx/subscriber 入队拒绝、`port_clear`/disconnect/release 归还和长时间运行预算不泄漏测试通过。
- 高频输入触发通知限速时，不产生无界通知风暴；限速只影响通知投递，不影响普通 `rx` 和未超限订阅者。
- release 与并发操作时，句柄立即不可用，资源锁和订阅记录清理一致。

## M5 TCP/UDP loopback

### 目标

在运行时核心稳定后接入真实异步网络 I/O，用 loopback 测试验证 transport 抽象和任务模型。

### 实现范围

- 实现 TCP client connect、read、write 和对端关闭处理。
- 按工具契约需要实现 TCP listen 模式时，补齐监听地址资源锁。
- 实现 UDP bind、send_to、recv_from 和释放后重绑。
- 实现 TCP/UDP loopback-only `port_scan`：解析单个 loopback 主机的端口或端口区间，执行目标、端口数量、并发、总超时和结果数量限制。
- 将 TCP/UDP I/O 错误映射到 07 的稳定错误码。
- 添加本地 echo server 或等价 loopback 测试辅助。

### 不做

- 不接入外部网络服务作为必测条件。
- 不实现 TLS、代理或复杂扫描策略。

### 验收门槛

- TCP client loopback 收发通过。
- UDP bind/send/recv loopback 通过。
- TCP/UDP `port_scan` 覆盖成功空结果、命中结果、`SCAN_TARGET_NOT_ALLOWED`、`SCAN_RANGE_TOO_LARGE`、`RESULT_TOO_LARGE` 和 `SCAN_TIMEOUT`。
- 地址冲突、连接超时、对端关闭、释放后重用路径符合 05、07、08。

## M6 串口 serialport 接入

### 目标

接入初版唯一串口库 `serialport`，并通过受控阻塞 worker 桥接到 Tokio 任务模型。

### 实现范围

- 实现 `port_scan` 的串口枚举路径，返回脱敏的候选串口摘要；网络扫描路径已在 M5 完成。
- 实现 `serial_config` 到 `serialport` builder 的配置映射。
- 实现串口打开、关闭、读、写和错误映射。
- 为每个串口实例建立 dedicated blocking task；短操作可使用受控 `spawn_blocking`。
- 通过控制消息、短超时读循环或关闭底层句柄促使阻塞读写退出。
- 确保 `instance_release(force=true)` 将资源锁转入 `closing` tombstone，后台关闭成功或超时后才允许同资源复用。

### 不做

- 不引入其他串口库作为默认或候选依赖。
- 不把真实串口硬件加入 CI 必需项。

### 验收门槛

- 串口配置参数校验和错误映射单元测试通过。
- 无硬件环境下自动化测试不失败。
- Windows 手工验收覆盖枚举、连接、回环收发、独占冲突、错误路径和释放重连。

## M7 MCP server 接入

### 目标

把已经可测试的应用服务接到 MCP stdio server，让初版工具契约可以通过真实 MCP 调用路径执行。

### 实现范围

- 使用 M0/M1 已复核并锁定的官方 Rust MCP SDK 能力，将 SDK 类型封装在 `mcp` 层；若 SDK 复核结论发生变化，必须先回写 09 和 M0/M1 记录再继续。
- 注册 03 中的全部初版工具。
- 将工具 handler 统一实现为：解析参数 -> 识别 session_id -> 调用 app service -> 包装成功或失败返回。
- 保证 MCP 层不直接操作 transport、队列或实例状态。
- 补齐 request_id、日志 span 和工具名关联。

### 不做

- 不实现 HTTP server 或 Web UI。
- 不让 SDK 类型扩散到 `app`、`runtime` 或 `transport` 层。

### 验收门槛

- 所有工具至少有一个 handler smoke test。
- MCP 调用返回外形与 07 一致。
- 会话绑定在不同 session_id 下隔离；实例相关工具省略 `handle_id` 时按当前会话默认实例解析并在成功返回中填充解析后的句柄。
- 缺少稳定 session_id 时，默认实例解析、`instance_use`、`port_subscribe_stream` 和 `port_unsubscribe_stream` 按 `SESSION_ID_UNAVAILABLE` 退化策略拒绝；显式 `handle_id` 且不依赖会话身份的工具仍可使用。
- 配置 `session_mode=single_dev` 仅在显式本地开发配置下启用；启用后所有相关工具返回和日志包含 `session_mode=single`，远程或多客户端配置下拒绝启用。

## M8 验收收敛与发布前整理

### 目标

按 08 的验收矩阵做最终收敛，确保初版功能、错误、并发、平台口径和文档一致。

### 实现范围

- 跑通单元测试、MockTransport 集成测试、TCP/UDP loopback、并发与多会话测试、错误模型测试。
- 执行 Windows 串口手工验收并记录结果。
- 检查日志字段、脱敏策略、`last_error` 摘要和恢复建议。
- 同步更新 11、12 或 README 中后续需要承接的运行配置和调试记录内容。
- 记录已知限制，例如 Linux/macOS 未抽测、某些串口驱动行为差异或 SDK 限制。

### 不做

- 不扩大初版范围。
- 不把 02 的进阶能力塞入发布前修补。

### 验收门槛

- 08 的进入编码后验收矩阵至少完成初版必测项。
- Windows 平台自动化和串口手工验收均有明确结论。
- 所有失败项都有阻塞/非阻塞分类、恢复建议和后续归属。

## M9 架构文件拆分与测试收敛

### 目标

在 M8 初版功能验收通过后，按 [04-运行时架构设计](04-运行时架构设计.md) 的推荐目录结构收敛物理文件布局，把当前折叠在少数大文件中的实现拆到职责明确的子模块中，同时保持工具可见行为、错误模型、状态机和依赖边界不变。

M9 是结构性收敛阶段，不改变 M0-M8 已完成事实，也不新增初版外能力。

### 实现范围

- 拆分 `mcp`、`app`、`model`、`runtime`、`transport` 和 `util` 顶层模块内的单文件实现。
- 将测试随被测职责迁移到相邻模块或保留在清晰的测试模块中，避免拆分后测试上下文失真。
- 保持 SDK Boundary Gate：`rmcp`、`RequestContext`、`CallToolResult`、`tool_router`、`tool_handler`、`schemars` 等 SDK 类型只允许出现在 `src/main.rs` 和 `src/mcp/*`。
- 每个拆分子任务完成时都必须记录回归命令证据，不能只以文件移动或函数占位作为完成依据。

### 不做

- 不改变 [03-初版工具契约](03-初版工具契约.md)、[05-状态机与资源生命周期](05-状态机与资源生命周期.md)、[06-并发与缓冲区设计](06-并发与缓冲区设计.md) 或 [07-错误模型与返回格式](07-错误模型与返回格式.md) 已确定的行为。
- 不修改 `mcp-server/`。
- 不新增 HTTP server、VISA、协议 helper、非 loopback scan allowlist、高级订阅或其他进阶能力。
- 不把真实串口硬件加入默认自动化测试前置。

### 验收门槛

- 每个 M9 子任务完成后，至少运行 `cargo fmt --check`、`cargo check`、`cargo test`。
- 每个 M9 子任务完成后，复核 SDK Boundary Gate 和 `mcp-server/` 未修改。
- 若拆分涉及日志、脱敏、错误模型或 MCP handler，额外运行对应 targeted tests，例如 `cargo test unit_redaction`、`cargo test m7_` 或相关模块测试。
- M9 总验收必须补施工记录或 acceptance 记录，说明拆分前后行为等价、测试通过和剩余结构限制。

## 推荐提交节奏

| 提交主题示例 | 覆盖阶段 | 说明 |
| --- | --- | --- |
| `chore: establish rust module skeleton` | M0 | 建立依赖和目录骨架。 |
| `feat(model): add response and error types` | M1 | 固化返回外形、错误类型和 ID。 |
| `feat(runtime): manage instances and resource locks` | M2 | 实例表、状态检查、资源锁。 |
| `feat(transport): add mock transport task loop` | M3 | MockTransport 和任务生命周期。 |
| `feat(runtime): implement queues buffers and subscriptions` | M4 | 队列、缓冲、订阅、多会话。 |
| `feat(transport): add tcp udp loopback support` | M5 | TCP/UDP 真实 I/O。 |
| `feat(transport): add serialport transport` | M6 | 串口接入与手工验收基础。 |
| `feat(mcp): expose initial tool handlers` | M7 | MCP server 和工具 handler。 |
| `test: complete initial acceptance matrix` | M8 | 验收补齐与发布前整理。 |
| `refactor: split architecture modules without behavior changes` | M9 | 按 04 推荐目录结构拆分文件并回归测试。 |

提交可以按实际开发拆得更细，但不建议把跨阶段大改混在同一提交里。例如不要同时改 MCP 协议层、串口阻塞 worker 和错误模型；这会让回归定位变困难。

## 阶段间依赖

```text
M0 工程骨架
-> M1 模型与错误
-> M2 实例生命周期
-> M3 MockTransport 任务骨架
-> M4 队列缓冲订阅
-> M5 TCP/UDP loopback
-> M6 串口 serialport
-> M7 MCP server 接入
-> M8 验收收敛
-> M9 架构文件拆分与测试收敛
```

可并行的工作有限：M1 完成后，部分文档、测试辅助和 MCP SDK 复核可以并行；但 M3 到 M6 的运行时路径应尽量顺序推进，因为后续 transport 都依赖前面已经稳定的任务、队列、错误和资源锁语义。

## 风险与调整规则

| 风险 | 调整方式 |
| --- | --- |
| 官方 Rust MCP SDK API 变化 | M0/M1 必须先做 spike 和能力记录；后续通过锁定版本、调整 `mcp` 层封装或拆小 handler 适配，仍必须使用官方 SDK。 |
| `serialport` 平台行为不一致 | 不回退到其他串口库；先封装平台差异、补手工验收记录，再决定是否延后某平台。 |
| 队列或状态机测试暴露设计冲突 | 回写 05、06 或 07，再继续实现，避免代码和文档分叉。 |
| TCP/UDP loopback 不稳定 | 先固定本地测试辅助和超时策略，不引入外部服务。 |
| 手工串口验收缺设备 | M6 可合入自动化部分，但 M8 发布结论必须标记串口手工验收未完成。 |
