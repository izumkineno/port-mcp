# 13 端口 MCP 代码施工大纲

本文档承接 [04-运行时架构设计](04-运行时架构设计.md)、[08-测试与验收计划](08-测试与验收计划.md)、[09-库选择与依赖决策](09-库选择与依赖决策.md) 与 [10-开发里程碑](10-开发里程碑.md)，把已经确定的架构、依赖和里程碑转化为面向代码落地的模块完成度检查表。

本文档不替代 [03-初版工具契约](03-初版工具契约.md)、[05-状态机与资源生命周期](05-状态机与资源生命周期.md)、[06-并发与缓冲区设计](06-并发与缓冲区设计.md) 和 [07-错误模型与返回格式](07-错误模型与返回格式.md)。具体工具参数、状态矩阵、缓冲策略和错误码仍以这些文档为准。

## 设计结论

- 初版代码施工按 `src/` 模块边界推进，每个模块必须先满足依赖边界和完成度检查，再让上层模块依赖。
- `model`、`util` 和错误返回是底座；`runtime` 是状态、实例、资源锁和任务生命周期的唯一运行时入口；`app` 只编排用例；`mcp` 只适配协议；`transport` 只处理底层 I/O。
- 任何模块不得为了赶进度绕过分层边界：MCP SDK 类型不能进入 `app`、`runtime` 或 `transport`；底层串口和 socket 句柄不能进入 `mcp` 或 `app`；松散 JSON 不能穿透服务边界。
- 施工完成度以可观察行为和测试证据判断，不以文件是否创建或函数是否占位判断。
- 每个模块的最小完成标准必须能映射到 [08-测试与验收计划](08-测试与验收计划.md) 中的自动化或手工验收项。

## 适用范围

| 范围 | 是否覆盖 | 说明 |
| --- | --- | --- |
| Rust 原生 MCP server | 覆盖 | 以 `src/mcp` 封装官方 Rust MCP SDK。 |
| Serial/TCP/UDP/Mock Transport | 覆盖 | 初版四类 transport 落点不同，进入任务层后收发语义统一。 |
| 实例生命周期与状态机 | 覆盖 | 以 `runtime` 和 `app` 共同落实，状态规则以 05 为准。 |
| 队列、缓冲区和订阅 | 覆盖 | 以 `runtime` 为主，通知适配由 `mcp` 承接。 |
| 错误模型与返回格式 | 覆盖 | 以 `model::error` 和 `mcp::response` 分层落实。 |
| VISA、协议 helper、心跳、异常注入 | 不覆盖 | 属于 [02-进阶实现大纲](02-进阶实现大纲.md)，不作为初版施工门槛。 |
| TypeScript MCP server | 不覆盖 | 当前初版目标是 Rust 原生 MCP server，TypeScript 目录不作为本施工大纲的执行对象。 |

## 推荐代码边界

初版代码结构沿用 [04-运行时架构设计](04-运行时架构设计.md) 的目录建议：

```text
src/
  main.rs
  mcp/
  app/
  runtime/
  transport/
  model/
  util/
```

依赖方向应保持单向：

```text
main
-> mcp
-> app
-> runtime
-> transport

app/runtime/transport
-> model
-> util
```

允许的横向依赖：

| 调用方 | 可依赖 | 规则 |
| --- | --- | --- |
| `mcp` | `app`、`model`、`util` | 只做协议入参解析、session 提取、返回包装和通知适配。 |
| `app` | `runtime`、`model`、`util` | 只编排工具用例和状态前置条件，不持有底层 I/O 句柄。 |
| `runtime` | `transport`、`model`、`util` | 管理实例、任务、队列、缓冲区、资源锁和统计。 |
| `transport` | `model`、`util` | 只打开、读写、关闭底层资源，并返回可映射的 transport 错误。 |
| `model` | `util` | 定义跨层稳定类型，不反向依赖业务服务。 |
| `util` | 标准库和轻量依赖 | 保持无业务状态。 |

禁止的反向依赖：

- `runtime`、`transport` 不引用 MCP SDK 类型。
- `model` 不引用 `app`、`runtime`、`transport` 或 MCP SDK。
- `app` 不直接引用 `serialport`、`tokio::net::TcpStream`、`UdpSocket` 或具体 transport 句柄。
- `mcp` 不直接修改实例状态、发送队列、接收缓冲区或资源锁。

## Todo 栏使用约定

后续每个模块和子模块的 `Todo 栏` 使用未勾选 Markdown 清单。实现者在施工时只能在对应代码、测试和验收证据已经落地后勾选，不应把“计划做”或“文件已创建”视为完成。

截至 M4 完成后，本文档的勾选含义为“对应能力已经有代码和测试证据”。若能力已落地但暂时折叠在单文件模块中，例如 `runtime::queues` 尚未拆成独立文件，仍可按能力完成勾选；若条目要求的真实 I/O、MCP 全工具接入或串口手工验收尚未进入当前阶段，则保持未勾选。

固定施工维度如下：

| 维度 | 含义 |
| --- | --- |
| 模型/接口 | 类型、trait、函数签名、入参和返回边界已经稳定。 |
| 实现 | 核心路径已经接入真实运行流，不只是占位。 |
| 错误/状态 | 错误映射、状态影响、资源清理和退化路径可解释。 |
| 测试 | 单元、集成、loopback、并发或手工验收有对应落点。 |
| 证据/验收 | 有命令、记录、日志、报告或文档引用支撑完成判断。 |

- `RuntimeRegistry` 是实例表和会话绑定的唯一入口。
- 全局锁只保护查找、插入、删除实例引用；底层 I/O 不在持锁状态执行。
- 单实例锁只覆盖短临界区状态修改、统计更新、缓冲区和任务句柄更新。
- 资源锁覆盖申请成功、冲突失败、连接失败回滚、断开释放、强制释放 `closing` tombstone、stale 诊断。
- `instance_release(force=true)` 后句柄立即失效，后台 close 不再通过旧句柄暴露可变状态。
- 发送队列 FIFO、队列满拒绝、单帧不交织、多实例互不阻塞。

- 动态预算和通知限速行为能被测试观察。
- 致命后台任务错误进入 `Error`，写入 `last_error`，并按 05 允许后续 query、clear、disconnect、release。

- 不在工具 handler 中直接修改 `RuntimeInstance` 内部字段。
- 不在持有实例状态锁时执行串口、TCP 或 UDP 读写。
- 不通过全局锁串行化所有实例 I/O。
- 不让订阅广播替代普通 rx buffer；二者必须同时存在。

### 完成检查

- 03 中全部初版工具都已注册，工具名稳定。
- 工具 handler 统一遵循 `parse args -> resolve session_id -> call app service -> map response`。
- SDK 类型只在 `mcp` 模块内出现，不扩散到 `app` 或更内层。
- 参数解析失败返回统一 `InvalidArgument` 结构，不把松散参数传入服务层。

- 不在 `mcp` handler 中直接改状态、资源锁、队列或缓冲区。
- 不在 `mcp` 中捕获错误后吞掉 category/code。
- 不让 SDK 的 session/context 缺口隐式改变多会话语义。
