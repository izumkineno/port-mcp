# 09 端口 MCP 库选择与依赖决策

本文档承接 [04-运行时架构设计](04-运行时架构设计.md)、[06-并发与缓冲区设计](06-并发与缓冲区设计.md)、[07-错误模型与返回格式](07-错误模型与返回格式.md) 与 [08-测试与验收计划](08-测试与验收计划.md)，把初版编码前需要确定的 Rust 库、依赖边界和替代方案收敛为可执行决策。

本文档不锁死具体 patch 版本，也不承诺某个外部 SDK 的最终 API。实现前仍需按本文的复核策略检查关键依赖的最新稳定版本、维护状态和 Windows 行为。

## 设计结论

- 初版依赖策略采用“少依赖、强封装、可替换”的方式：优先使用 Rust 标准库、Tokio 生态和成熟小库，不引入重型框架。
- 异步运行时统一使用 `tokio`，TCP/UDP 直接使用 `tokio::net`。
- 串口使用 `serialport`；通过 dedicated blocking task 或受控 `spawn_blocking` 封装接入 Tokio，保持上层统一的 Transport 读写任务模型。
- MCP 协议层必须使用官方 Rust MCP SDK；SDK 类型只能封装在 `mcp` 层，不能让 SDK 细节污染应用服务层。SDK 可用性复核是 M0/M1 的阻塞门槛，而不是 M7 才发现的集成事项。
- 返回结构、配置和模型序列化使用 `serde` 与 `serde_json`。
- 领域错误使用 `thiserror`，启动、边界 glue 和不可恢复内部上下文可使用 `anyhow`。
- 日志与诊断使用 `tracing` 和 `tracing-subscriber`，并遵守 [07-错误模型与返回格式](07-错误模型与返回格式.md) 的脱敏规则。
- 缓冲区数据优先使用 `bytes`；任务通信使用 `tokio::sync` 中的 `mpsc`、`oneshot`、`watch` 或等价轻量原语。
- 自动化测试优先依赖 `tokio::test`、自研 `MockTransport` 和本地 TCP/UDP loopback；可选引入 `rstest` 做参数矩阵。

## 选择原则

| 原则 | 说明 |
| --- | --- |
| 与 Tokio 兼容 | 端口 I/O、后台任务、队列和取消信号已经由 04 确定运行在同一个 Tokio runtime 中。 |
| 跨平台优先 | 初版 Windows 必测，Linux/macOS 可选抽测；串口库必须优先复核 Windows 行为。 |
| 封装外部变化 | MCP SDK、串口库和日志实现只暴露在局部模块内，应用层只依赖本项目模型。 |
| 可测试 | CI 不依赖硬件，所有核心路径必须能通过 MockTransport 或 loopback 测试。 |
| 少依赖 | 不为初版暂不实现的协议、UI、数据库或 actor 模型提前引入依赖。 |
| 错误友好 | 库错误必须能映射为 07 定义的 category、code、retryable、details 和日志字段。 |

## 推荐依赖总表

| 领域 | 推荐候选 | 初版用法 | 备注 |
| --- | --- | --- | --- |
| 异步运行时 | `tokio` | runtime、任务、超时、TCP/UDP、channel。 | 必选。启用的 feature 以实际代码最小集合为准。 |
| MCP 协议层 | 官方 Rust MCP SDK | stdio server、工具注册、请求解析、响应包装。 | 必选；实现前复核官方 SDK 版本、API 和 stdio 支持，并通过 `mcp` 层隔离。 |
| 序列化 | `serde`、`serde_json` | 入参、返回结构、错误 details、调试记录草案。 | 必选。 |
| JSON schema | `schemars` | 工具 schema 或文档生成。 | 可选；只有 MCP SDK 或客户端配置需要时再引入。 |
| 错误类型 | `thiserror` | `PortMcpError`、Transport error、状态错误。 | 必选。 |
| 错误上下文 | `anyhow` | `main`、启动流程、测试辅助、一次性内部 glue。 | 可选；不作为领域错误返回类型。 |
| 日志追踪 | `tracing`、`tracing-subscriber` | request_id、handle_id、state、error_code、任务退出日志。 | 必选。 |
| 时间 | `time` 或 `chrono` | `timestamp`、验收记录、日志字段。 | 推荐优先评估 `time`，保持依赖轻量。 |
| ID | `uuid` 或自增/随机短 ID | `handle_id`、`request_id`。 | 初版可自研生成器；需要跨进程唯一性时再引入 `uuid`。 |
| 字节缓冲 | `bytes` | rx buffer、payload 切片、发送帧。 | 推荐。 |
| TCP/UDP | `tokio::net` | TCP client/listen、UDP bind/send/recv、loopback 测试。 | 必选。 |
| 串口 | `serialport` | 阻塞 serial transport，经专用任务桥接到 async。 | 初版默认方案，需实现前复核 Windows 行为。 |
| 测试矩阵 | `tokio::test`、可选 `rstest` | 异步测试、状态矩阵、参数矩阵。 | `rstest` 只在重复矩阵明显增多时引入。 |

## MCP 协议层选择

MCP 接入层只负责 server 启动、工具注册、参数解析、会话上下文识别和统一返回包装。库选择必须服务于这个边界。

推荐策略：

1. 在 M0/M1 先复核官方 Rust MCP SDK 的 crate 名称、稳定版本、许可证、stdio server、tool schema、request context、resource notification、error response 和 session/context 支持，并提交最小 spike 记录。
2. 初版必须在 `mcp::server` 和 `mcp::tools` 中使用官方 SDK，不手写协议层替代实现。
3. 如果官方 SDK API 快速变化，只允许通过锁定版本、调整 `mcp` 层封装或缩小 SDK 暴露面来处理；若缺少稳定 session_id，必须按 04 的 `session_mode=unavailable` 退化策略禁用依赖会话隔离的工具。
4. 应用服务层、运行时核心层和 Transport 层不得直接引用 SDK 类型。

这样可以确保官方 SDK 后续升级时，只影响 `mcp` 目录，不影响 [04-运行时架构设计](04-运行时架构设计.md) 中定义的 `app`、`runtime` 和 `transport` 分层。

## 异步运行时与任务原语

初版统一使用 `tokio`，不再并行引入其他异步 runtime。

| 能力 | 推荐原语 | 用法 |
| --- | --- | --- |
| 后台任务 | `tokio::spawn` | 连接后的发送任务、接收任务、订阅通知任务。 |
| 超时 | `tokio::time::timeout` | connect、read、write、scan 和断开等待。 |
| 发送队列 | `tokio::sync::mpsc` | 每实例 `tx_queue`，满时拒绝新发送。 |
| 任务退出通知 | `oneshot` 或 join handle | 断开、释放和致命错误收敛。 |
| 状态或取消通知 | `watch` 或内部控制消息 | disconnect/release 触发后台任务退出。 |
| 共享状态 | `Arc` 加短临界区锁 | registry、实例状态、统计和缓冲区。 |

任务原语的选择必须满足 [05-状态机与资源生命周期](05-状态机与资源生命周期.md) 的要求：`instance_release(force=true)` 后句柄立即失效，资源锁进入 `closing` tombstone 隔离，同资源在后台 close 完成或超时前不得复用。

## Serial/TCP/UDP Transport 库选择

### TCP/UDP

TCP 与 UDP 不需要额外网络库，直接使用 `tokio::net`。

| 类型 | 推荐实现 | 说明 |
| --- | --- | --- |
| TCP client | `tokio::net::TcpStream` | 连接本地或远端设备，支持 timeout 和读写拆分。 |
| TCP listen | `tokio::net::TcpListener` | 仅在初版工具契约需要 listen 模式时启用，资源锁 key 为 `tcp-listen:<ip>:<port>`。 |
| UDP | `tokio::net::UdpSocket` | bind、send_to、recv_from，资源锁 key 为 `udp-bind:<ip>:<port>`。 |

TCP/UDP loopback 测试直接基于 Tokio 构建本地 echo server，不引入测试专用网络框架。

### Serial

串口是初版依赖风险最高的部分，必须通过 `transport::serial` 封装。

默认方案：

| 方案 | 用法 | 风险 | 缓解 |
| --- | --- | --- | --- |
| `serialport` + dedicated blocking task | 每个串口实例拥有受控阻塞读写 worker，通过 channel 与 Tokio 任务交互。 | 需要自行处理阻塞读取消、任务退出和 backpressure。 | 使用控制消息、短超时读循环、关闭底层句柄促退，并让资源锁在 force release 时进入 `closing` tombstone。 |
| `serialport` + 受控 `spawn_blocking` | 只用于打开、配置或短操作；长读写循环必须有明确生命周期控制。 | 长读循环不适合无控制地占用 blocking pool。 | 长生命周期 I/O 优先使用 dedicated blocking task，避免占满 Tokio blocking pool。 |

无论采用哪个方案，上层只看统一读写句柄和 TransportTask，不直接感知串口库类型。串口打开失败、权限不足、设备不存在和占用冲突必须映射到 [07-错误模型与返回格式](07-错误模型与返回格式.md) 定义的错误类别和 code。

## 缓冲区与队列

初版缓冲区和队列实现应保持简单、可观察、可测试。

| 能力 | 推荐选择 | 规则来源 |
| --- | --- | --- |
| 发送队列 | `tokio::sync::mpsc` | [06-并发与缓冲区设计](06-并发与缓冲区设计.md) 的 tx 满拒绝。 |
| 普通 rx buffer | `VecDeque<Bytes>` 或等价结构 | rx 满丢旧保新，累计 dropped 统计。 |
| 订阅者队列 | 每订阅者独立 `mpsc` | 慢订阅者只丢自己的旧通知。 |
| payload | `bytes::Bytes` / `BytesMut` | 减少复制，便于截断和统计。 |
| 统计 | 本项目自定义结构 | 记录 sent、received、dropped、last_activity。 |

暂不引入重型 actor 框架或流处理框架。当前队列语义已经足够覆盖初版并发要求，额外框架会增加错误映射、取消语义和测试复杂度。

## 序列化、返回结构与时间 ID

`serde` 和 `serde_json` 是初版返回结构和入参解析的基础依赖。所有工具成功和失败返回都应由强类型结构序列化得到，不在 handler 中拼接松散 JSON。

时间和 ID 选择建议：

| 能力 | 初版建议 | 说明 |
| --- | --- | --- |
| `timestamp` | `time` 或 `chrono` 输出 RFC3339 | 推荐优先评估 `time`，减少依赖面。 |
| `request_id` | 自研递增或短随机 ID | 必须能贯穿工具返回与日志。 |
| `handle_id` | 自研生成器，必要时可用 `uuid` | 初版只要进程内唯一且不易混淆即可。 |
| schema | 可选 `schemars` | 只有当 MCP SDK 或客户端配置需要 schema 生成时引入。 |

## 错误与日志

领域错误统一由本项目错误类型表达，推荐使用 `thiserror` 派生。

边界规则：

- `PortMcpError` 或等价领域错误必须携带 category、code、message、recovery_hint、retryable 和 details 所需信息。
- `anyhow` 只能用于启动流程、测试辅助、内部 glue 或临时上下文，不应跨过应用服务边界成为工具错误返回。
- 外部库错误进入工具返回前必须转换为稳定 code，例如 `SERIAL_OPEN_TIMEOUT`、`WRITE_IO_FAILED`、`TRANSPORT_CLOSED`。
- 日志使用 `tracing` 记录内部细节，工具返回默认脱敏。
- 日志字段至少应支持 `request_id`、`tool`、`handle_id`、`session_id`、`state`、`error.category`、`error.code` 和耗时。

日志 subscriber 初版可使用 `tracing-subscriber` 输出 stderr 或文件，具体格式留给 [11-配置与运行说明](11-配置与运行说明.md) 和 [12-调试记录格式](12-调试记录格式.md) 承接。

## 测试依赖

初版测试依赖必须服务于 [08-测试与验收计划](08-测试与验收计划.md)，不能把真实硬件带入 CI。

| 测试类型 | 推荐工具 | 说明 |
| --- | --- | --- |
| 异步单元测试 | `tokio::test` | 覆盖状态矩阵、超时、任务退出和 channel 行为。 |
| 参数矩阵 | 可选 `rstest` | 状态和错误码矩阵较多时再引入。 |
| MockTransport | 本项目自研 | 一等测试基座，不作为用户可见连接类型。 |
| TCP/UDP loopback | `tokio::net` 自建 echo | 无需外部服务和额外框架。 |
| 临时文件或目录 | 标准库或轻量 dev-dependency | 只有调试记录测试需要时再引入。 |

测试中应优先断言稳定行为：状态、错误 category/code、资源锁释放、队列溢出策略、订阅隔离和脱敏结果。不要断言外部库原始错误文本。

## 暂不引入的库或能力

以下依赖或能力不纳入初版默认依赖：

| 类型 | 不引入原因 |
| --- | --- |
| VISA / USBTMC / GPIB 库 | 属于 02 中的进阶硬件能力，不阻塞 Serial/TCP/UDP 初版。 |
| Modbus / SCPI / SLIP helper | 初版只做字节传输，不做协议语义解析。 |
| 重型 actor 框架 | 06 的队列和任务模型可由 Tokio 原语完成。 |
| Web 框架 | 初版 MCP 通过 stdio 接入，不需要 HTTP server。 |
| 数据库 | 实例、状态和缓冲区均为进程内运行时数据。 |
| 配置框架大礼包 | 初版配置项少，可先用 serde + 简单文件或环境读取。 |
| 高级日志导出栈 | 调试记录格式尚未固化，先保留 tracing 事件。 |

## 版本锁定与复核策略

- 使用 `Cargo.lock` 固定可复现构建。
- `Cargo.toml` 不使用 wildcard 版本。
- 每个依赖只开启实际需要的 feature。
- `tokio`、官方 MCP SDK、`serialport`、`tracing`、`serde` 等关键依赖在实现前需要复核最新稳定版本、维护状态、许可证和 Windows 支持。
- 如果首选依赖复核失败，优先在对应模块内部切换替代方案，不改变 03 到 08 已确定的工具契约、状态机和错误模型。
- 新增依赖必须能说明它服务的文档条款或测试用例。

## 风险与替代方案

| 风险 | 影响 | 默认应对 |
| --- | --- | --- |
| 官方 Rust MCP SDK API 变化 | 工具注册和响应包装可能返工。 | SDK 类型封装在 `mcp` 层；通过锁定版本、调整封装或拆小 handler 适配，不手写协议层替代实现。 |
| `serialport` 阻塞 I/O 与 Tokio 生命周期桥接不当 | 串口连接、取消、超时或 release 行为不稳定。 | 使用 dedicated blocking task 或受控 `spawn_blocking`，并把生命周期封装在 `transport::serial` 内。 |
| 阻塞串口读难以取消 | force release 后后台任务退出变慢。 | 通过控制消息、短超时读循环或关闭底层句柄促退；句柄同步失效，资源锁同步转入 `closing` tombstone，后台 close 成功或 deadline 后才释放 tombstone。 |
| 外部库错误文本含敏感信息 | 工具返回可能泄漏路径、用户名或设备信息。 | 在错误映射层做脱敏，原始错误只进入受控日志。 |
| 依赖 feature 过宽 | 构建变慢，跨平台差异增加。 | 最小 feature 启用，新增 feature 需对应具体功能。 |

- [11-配置与运行说明](11-配置与运行说明.md) 应承接日志输出、MCP 客户端配置、串口权限和平台差异。
- [12-调试记录格式](12-调试记录格式.md) 应承接 tracing 字段、收发帧记录、错误链和手工验收记录。
- 如果后续编码发现依赖选择需要调整，应优先更新本文，再同步影响到里程碑、配置说明和测试计划。

## 设计验收标准

本文档完成后，进入初版编码前至少应满足：

- 已明确初版必选依赖、可选依赖和暂不引入依赖。
- 已给出 MCP 协议层、Tokio runtime、Serial/TCP/UDP、错误日志、序列化、缓冲队列和测试依赖的默认方案。
- 已明确 MCP 协议层必须使用官方 Rust MCP SDK，并明确串口默认使用 `serialport` 的 Tokio 桥接策略。
- 已说明外部库错误如何收敛到 07 的稳定错误模型。
- 已说明 CI 不依赖真实硬件，串口测试仍按 08 的 Windows 手工验收执行。
- 已把原计划中的开发里程碑、配置运行说明和调试记录格式顺延到 10、11、12。
