# 11 端口 MCP 配置与运行说明

本文档承接 [04-运行时架构设计](04-运行时架构设计.md)、[06-并发与缓冲区设计](06-并发与缓冲区设计.md)、[07-错误模型与返回格式](07-错误模型与返回格式.md)、[08-测试与验收计划](08-测试与验收计划.md)、[09-库选择与依赖决策](09-库选择与依赖决策.md) 与 [10-开发里程碑](10-开发里程碑.md)，说明初版工程如何在本地运行、如何被 MCP 客户端以 stdio 方式拉起、如何配置日志与容量限制，以及不同平台使用串口时需要注意的权限和验收准备。

本文档只定义运行和配置口径，不替代具体实现代码、发布脚本或调试记录格式。收发帧、错误链、验收截图和设备调试记录由后续 [12-调试记录格式](12-调试记录格式.md) 承接。

## 设计结论

- 初版作为 Rust 原生 MCP server 运行，通过官方 Rust MCP SDK 暴露 stdio server 和工具注册能力。
- MCP 客户端配置只应调用编译后的服务进程或 `cargo run` 入口，不需要额外 TypeScript 协议壳。
- MCP 协议层必须使用官方 Rust MCP SDK；SDK 类型只允许封装在 `mcp` 层，运行时、应用服务和 Transport 层不直接依赖 SDK 类型。
- 本地开发优先使用 `cargo run` 和 TCP/UDP loopback 验证无硬件路径；CI 不依赖真实串口硬件。
- 串口使用 `serialport`，通过 dedicated blocking task 或受控 `spawn_blocking` 桥接到 Tokio；真实串口能力通过 Windows 手工验收确认。
- 日志默认输出结构化、脱敏的 `tracing` 事件；工具返回中的 verbose 错误详情默认关闭。
- 容量限制默认采用 [06-并发与缓冲区设计](06-并发与缓冲区设计.md) 的建议值，后续即使允许配置覆盖，也必须保留硬上限。

## 运行环境

初版默认运行环境如下：

| 项目 | 初版要求 | 说明 |
| --- | --- | --- |
| Rust toolchain | 稳定版 Rust | 具体版本在实现阶段锁定；构建应由 `Cargo.lock` 保持可复现。 |
| 异步运行时 | `tokio` | 所有后台任务、TCP/UDP、队列、超时和取消信号统一运行在 Tokio runtime 内。 |
| MCP 协议层 | 官方 Rust MCP SDK | 用于 stdio server、工具注册、请求解析和响应包装。 |
| TCP/UDP | `tokio::net` | 本地 loopback 是自动化验证的一等路径。 |
| 串口 | `serialport` | 真实硬件不进入 CI；Windows 手工验收是初版发布前门槛。 |
| 日志 | `tracing`、`tracing-subscriber` | 默认输出脱敏结构化事件。 |
| 平台 | Windows 必测，Linux/macOS 可选抽测 | Linux/macOS 主要验证串口路径、权限和 loopback 行为。 |

## 本地开发运行方式

初版实现完成后，本地开发建议保留两种运行方式。

### 直接运行 server

```powershell
cargo run
```

该方式用于让 MCP 客户端或人工测试通过 stdio 与 server 通信。服务启动时应完成以下动作：

1. 读取运行配置和默认限制。
2. 初始化 `tracing` subscriber。
3. 创建 `RuntimeRegistry`。
4. 创建官方 Rust MCP SDK 的 stdio server。
5. 注册 [03-初版工具契约](03-初版工具契约.md) 中的初版工具。
6. 阻塞运行 stdio server，直到客户端关闭或进程退出。

### 本地验证无硬件路径

```powershell
cargo test
```

自动化测试应默认覆盖单元测试、MockTransport、TCP/UDP loopback、并发、多会话和错误模型。测试不得要求连接真实串口设备；串口相关自动化只验证参数校验、错误映射、资源锁和可替代的内部模拟路径。

## MCP 客户端配置

MCP 客户端应以 stdio 方式启动本服务。配置字段名称会随客户端不同而变化，但语义应保持一致：

| 字段语义 | 建议值 | 说明 |
| --- | --- | --- |
| server name | `port-mcp` | 客户端侧展示名。 |
| command | 编译后的可执行文件路径，或开发期 `cargo` | 发布使用可执行文件；开发期可使用 `cargo run`。 |
| args | 开发期为 `run` 或 `run --manifest-path <path>` | 发布包可为空，具体以打包方式为准。 |
| transport | `stdio` | 初版不要求 HTTP server。 |
| cwd | 工程目录或发布目录 | 开发期使用仓库根目录；发布后使用可执行文件所在目录。 |
| env | 日志级别、配置文件路径等 | 不应通过环境变量传入敏感密钥；初版也不需要认证信息。 |

开发期 MCP 客户端配置可按以下语义组织：

```json
{
  "name": "port-mcp",
  "command": "cargo",
  "args": ["run"],
  "cwd": "<repo-root>",
  "transport": "stdio",
  "env": {
    "PORT_MCP_LOG": "info"
  }
}
```

发布后建议改为直接调用编译产物：

```json
{
  "name": "port-mcp",
  "command": "<path-to-port-mcp-executable>",
  "args": [],
  "transport": "stdio",
  "env": {
    "PORT_MCP_LOG": "info"
  }
}
```

客户端连通后，最小 smoke test 应覆盖：

1. 调用 `instance_create` 创建一个 TCP、UDP 或 serial 实例。
2. 调用 `instance_list` 确认实例可见。
3. 调用 `instance_query` 确认返回包含 `ok`、`tool`、`request_id`、`timestamp` 和实例状态。
4. 在 loopback 或真实串口准备好时继续执行配置、连接、发送、读取和释放路径。

## 配置项总表

初版配置可以来自环境变量、简单配置文件或后续发布脚本，但应用层应统一转换为强类型运行配置。下表定义配置语义，不强制具体读取实现。

| 配置项 | 默认值 | 可配置性 | 说明 |
| --- | --- | --- | --- |
| `log_level` | `info` | 可配置 | 支持 `error`、`warn`、`info`、`debug`、`trace`。 |
| `log_format` | `json` 或结构化文本 | 可配置 | 初版至少保证字段稳定，格式以实现复核为准。 |
| `verbose_error_details` | `false` | 可配置但默认关闭 | 即使开启，也不得返回环境变量值、认证信息或未裁剪大 payload。 |
| `session_mode` | `auto` | 可配置，仅允许 `auto`、`unavailable`、`single_dev` | `single_dev` 仅允许 stdio 本地开发入口显式启用；启动日志必须 warn，工具返回和日志必须标记 `session_mode=single`，网络 transport、远程入口、第二客户端或等价多会话入口必须拒绝。 |
| `tx_queue_max_items` | `256` | 可配置 | 每实例发送队列长度。 |
| `tx_frame_max_bytes` | `64 KiB` | 可配置 | 单次发送帧上限。 |
| `rx_buffer_max_bytes` | `1 MiB` | 可配置 | 每实例普通 `rx` 缓冲区容量。 |
| `pull_default_max_bytes` | `4 KiB` | 可配置 | `port_pull` 未传 `max_bytes` 时的默认返回上限。 |
| `pull_max_bytes` | `64 KiB` | 可配置 | 单次 `port_pull` 硬限制以内的可返回上限。 |
| `subscriber_queue_max_items` | `128` | 可配置 | 每订阅者通知队列长度。 |
| `subscriber_payload_max_bytes` | `16 KiB` | 可配置 | 单次订阅通知 payload 上限。 |
| `subscriber_notifications_per_sec` | `64` | 可配置 | 单订阅者每秒通知投递上限。 |
| `instance_notifications_per_sec` | `256` | 可配置 | 单实例所有订阅者合计每秒通知投递上限。 |
| `global_notifications_per_sec` | `1024` | 可配置 | 全进程所有订阅者合计每秒通知投递上限。 |
| `notification_burst` | `128` | 可配置 | 通知限速 token bucket 或等价窗口的短时突发容量。 |
| `scan_max_concurrency` | `64` | 可配置 | 网络扫描最大并发探测数。 |
| `scan_max_ports` | `256` | 可配置 | 单次网络扫描允许探测的最大端口数。 |
| `scan_total_timeout_ms` | `10000` | 可配置 | 单次网络扫描整体超时上限。 |
| `scan_allowed_hosts` | `127.0.0.0/8, ::1` | 初版固定 loopback-only | 网络扫描允许目标；初版保留该字段用于文档表达，但不得配置为非 loopback。非 loopback allowlist 属于进阶能力。 |
| `max_instances` | `64` | 可配置 | 单进程最多未释放实例数。 |
| `max_subscribers_per_instance` | `32` | 可配置 | 单实例最多订阅者数量。 |
| `max_total_buffer_bytes` | `128 MiB` | 可配置 | 全进程普通 `rx` 缓冲区组合预算硬上限。 |
| `max_total_queued_bytes` | `64 MiB` | 可配置 | 全进程发送队列和订阅队列组合预算硬上限。 |
| `force_close_deadline_ms` | `5000` | 可配置，范围 `1000..30000` | 强制释放后资源锁 `closing` tombstone 的后台关闭期限；`0`、负数或超出范围必须拒绝启动或返回 `InvalidArgument/INVALID_RANGE`。 |
| `connect_timeout_ms` | 实现阶段确定 | 可配置 | TCP connect、串口打开和通用连接超时。 |
| `read_timeout_ms` | 实现阶段确定 | 可配置 | `port_pull` 等待读取时的默认超时。 |
| `serial_read_poll_timeout_ms` | 实现阶段确定 | 可配置 | `serialport` 阻塞 worker 的短超时读循环间隔。 |

所有容量类配置都必须经过范围校验。超出允许范围时，启动配置错误或工具参数错误应映射到 [07-错误模型与返回格式](07-错误模型与返回格式.md) 中的 `InvalidArgument/INVALID_RANGE` 或等价启动错误。初版不允许通过配置把 `scan_allowed_hosts` 扩展到非 loopback；`0.0.0.0`、`::`、通配主机名、CIDR 私网/公网、链路本地、多播和 DNS 展开目标都不得作为扫描目标。

`session_mode=single_dev` 是受控单会话开发模式，不得作为共享、远程或生产运行配置默认值。初版仅 stdio 本地开发入口允许启用；检测到网络 transport、远程入口、第二客户端或等价多会话入口时，必须启动失败或拒绝该会话，而不是仅记录 warning。启用时必须在工具返回、日志和调试记录中持续标记 `session_mode=single`。

## 默认限制与硬上限

默认限制来自 [06-并发与缓冲区设计](06-并发与缓冲区设计.md)：

| 限制项 | 默认值 | 触发后的行为 |
| --- | --- | --- |
| `tx_queue_max_items` | `256` | 新 `port_send` 被拒绝，返回 `BufferLimitExceeded/TX_QUEUE_FULL`。 |
| `tx_frame_max_bytes` | `64 KiB` | 当前发送被拒绝，返回 `BufferLimitExceeded/TX_FRAME_TOO_LARGE`。 |
| `rx_buffer_max_bytes` | `1 MiB` | 普通 `rx` 丢旧保新，累计 `rx_dropped_bytes`。 |
| `pull_default_max_bytes` | `4 KiB` | 未指定 `max_bytes` 时按该值截取。 |
| `pull_max_bytes` | `64 KiB` | 超限请求返回 `BufferLimitExceeded/PULL_MAX_BYTES_EXCEEDED`。 |
| `subscriber_queue_max_items` | `128` | 慢订阅者只丢自己的旧通知。 |
| `subscriber_payload_max_bytes` | `16 KiB` | 超限订阅请求返回 `BufferLimitExceeded/SUBSCRIBER_PAYLOAD_TOO_LARGE`。 |
| `subscriber_notifications_per_sec` | `64` | 超限时合并或丢弃该订阅者通知，累计 `dropped_notifications`。 |
| `instance_notifications_per_sec` | `256` | 超限时对该实例订阅通知执行合并或丢弃，不影响普通 `rx`。 |
| `global_notifications_per_sec` | `1024` | 超限时对全进程通知投递执行合并或丢弃，限速日志必须节流。 |
| `notification_burst` | `128` | 允许短时突发；超过 burst 后按限速策略合并或丢弃。 |
| `force_close_deadline_ms` | `5000` | 低于 `1000`、高于 `30000`、`0` 或负数配置必须拒绝；deadline 到达后按 `RESOURCE_LOCK_STALE` 或具体 busy code 处理。 |
| `scan_max_concurrency` | `64` | 超限扫描返回 `BufferLimitExceeded/SCAN_RANGE_TOO_LARGE`。 |
| `scan_max_ports` | `256` | 超限扫描返回 `BufferLimitExceeded/SCAN_RANGE_TOO_LARGE`。 |
| `scan_total_timeout_ms` | `10000` | 整体扫描超时返回 `ConnectTimeout/SCAN_TIMEOUT`。 |
| `max_instances` | `64` | 超限创建返回容量类错误或启动配置错误。 |
| `max_subscribers_per_instance` | `32` | 超限订阅返回 `BufferLimitExceeded/SUBSCRIBER_LIMIT_EXCEEDED`。 |
| `max_total_buffer_bytes` | `128 MiB` | 运行时 `buffer_bytes_budget` reserve 失败时触发丢旧保新或丢弃统计。 |
| `max_total_queued_bytes` | `64 MiB` | 运行时 `queued_bytes_budget` reserve 失败时拒绝新 tx 或按慢订阅者策略丢旧通知。 |

硬上限应由实现集中定义在 `RuntimeLimits` 或等价结构中，至少包含 `ABS_MAX_TOTAL_BUFFER_BYTES` 和 `ABS_MAX_TOTAL_QUEUED_BYTES` 两个绝对上限。配置文件和环境变量只能在绝对上限以内调小或调高默认值，不能允许一次工具调用占用不可控内存。初版采用运行时动态扣账：`rx_buffer`、tx frame、订阅通知 payload 在入队或追加前 reserve 全局预算，在发送完成、通知丢弃、`port_clear`、disconnect 或 release 时归还预算。默认值不要求按最坏情况静态乘积通过，但必须通过动态预算测试证明不会超过 `max_total_buffer_bytes` 和 `max_total_queued_bytes`。

监听地址与扫描目标必须分离：`tcp_udp_config` 可以在用户显式配置时绑定 `0.0.0.0` 或 `::` 作为 listen 地址；`port_scan` 永远不得把 `0.0.0.0`、`::`、通配地址或接口地址展开为扫描目标。

## 日志与诊断配置

日志事件必须能支持工具调用和后台任务排查。初版至少应覆盖以下字段：

| 字段 | 说明 |
| --- | --- |
| `event` | `tool_error`、`state_error`、`task_error`、`resource_lock_error`、`background_close_error` 等。 |
| `timestamp` | UTC 时间。 |
| `request_id` | 关联单次工具调用。 |
| `tool` | 工具名。 |
| `session_id` | 脱敏后的 MCP 会话标识或摘要。 |
| `handle_id` | 实例句柄。 |
| `state_before` / `state_after` | 状态变化前后。 |
| `error.category` / `error.code` | 稳定错误类别和细分错误码。 |
| `resource_key` | 内部规范化资源 key，按脱敏规则输出。 |
| `duration_ms` | 工具调用或后台操作耗时。 |
| `sensitive` | 当前事件是否含敏感字段。 |

日志级别建议：

| 级别 | 用途 |
| --- | --- |
| `error` | 工具失败、致命后台任务退出、断开释放失败。 |
| `warn` | 可恢复异常、慢订阅者丢弃、资源锁疑似残留。 |
| `info` | server 启动、工具注册、连接建立、正常断开和释放摘要。 |
| `debug` | 参数摘要、状态转移、队列长度、缓冲区统计。 |
| `trace` | 更细的任务循环事件；不得默认开启。 |

工具返回默认只包含脱敏后的错误详情。完整 cause chain、底层 OS 错误文本和后台 close 结果只能进入受控日志，并按 [07-错误模型与返回格式](07-错误模型与返回格式.md) 标记敏感性。

## 串口权限与平台差异

串口配置和运行必须按平台处理命名、权限和设备占用差异。

| 平台 | 串口命名示例 | 常见问题 | 处理建议 |
| --- | --- | --- | --- |
| Windows | `COM3`、`COM10` | 设备被串口调试助手占用、驱动未安装、USB 设备重插后端口变化。 | 在设备管理器确认端口名；关闭占用程序；重插后重新 `port_scan`。 |
| Linux | `/dev/ttyUSB0`、`/dev/ttyACM0` | 当前用户无权限、设备组不同、udev 规则缺失。 | 将用户加入合适设备组或配置 udev；重新登录后再测试。 |
| macOS | `/dev/tty.usbserial-*`、`/dev/tty.usbmodem*` | 设备路径变化、系统权限提示、驱动兼容差异。 | 使用扫描结果选择端口；确认系统隐私与驱动状态。 |

串口连接失败时，工具返回不得暴露完整本地路径、用户名、环境变量或未裁剪的 OS 错误文本。推荐返回 `io_kind`、脱敏资源摘要、`recovery_hint` 和下一步工具调用建议。

## TCP/UDP loopback 运行说明

TCP/UDP loopback 是无硬件路径的首选验证方式。初版测试或本地开发辅助应覆盖：

- TCP client 连接本地 echo server，`port_send` 后可以 `port_pull` 到回显。
- UDP bind 后向本地 echo 端发送 datagram，并能收到相同 payload。
- TCP listen 地址和 UDP bind 地址冲突可返回结构化 `ResourceBusy` 错误。
- 断开或释放后，相同地址端口可重新使用。
- 连接超时、空读超时和对端关闭均能映射到稳定错误码。

loopback 测试不得依赖外部网络服务。测试端口应由测试辅助动态选择，避免固定端口导致偶发冲突。

## Windows 串口手工验收准备

真实串口能力在发布或进入真实设备调试前必须完成 Windows 手工验收。准备清单如下：

| 项目 | 要求 |
| --- | --- |
| 设备 | 真实串口设备、USB-TTL 回环或虚拟串口对三选一。 |
| 驱动 | 系统可枚举目标串口，设备管理器中能看到稳定 `COMx` 名称。 |
| 占用检查 | 关闭其他串口调试助手、厂商工具或旧 server 进程。 |
| 回环路径 | USB-TTL 回环需连接 TX/RX；虚拟串口对需确认成对端口可互通。 |
| 日志 | `PORT_MCP_LOG=debug` 或等价配置，便于记录连接、收发和错误摘要。 |
| 记录 | 保存平台版本、设备方案、串口名、工具调用摘要、失败日志摘要和最终结论。 |

最低验收路径：

1. `port_scan` 能发现目标串口。
2. `instance_create` 创建 serial 实例。
3. `serial_config` 写入 baudrate、data bits、parity、stop bits 和 timeout。
4. `port_connect` 后状态进入 `Connected`。
5. `port_send` 发送 text 和 hex 样例。
6. `port_pull` 能在回环或设备响应中读到预期字节。
7. 同一串口第二个 connected 实例返回 `ResourceBusy/SERIAL_PORT_BUSY`。
8. 不存在串口、权限不足或被占用时返回结构化错误。
9. `port_disconnect` 和 `instance_release` 后资源可重新连接。

## 安全与脱敏默认

初版运行配置必须默认安全：

- 不在工具返回中暴露系统用户名、完整本地路径、环境变量值、堆栈或未裁剪 payload。
- 不在日志中记录认证信息或环境变量值；如某事件含敏感上下文，必须标记 `sensitive=true`。
- 串口名、IP 和端口可作为资源摘要返回，但底层错误文本需要裁剪。
- `verbose_error_details=false` 是默认值，打开后仍不得突破脱敏底线。
- MCP 客户端配置不需要密钥；如后续增加远程传输或认证，应另行补充安全设计。

## 常见失败与恢复建议

| 现象 | 可能原因 | 建议动作 |
| --- | --- | --- |
| MCP 客户端无法启动 server | `command` 路径错误、`cwd` 不正确、可执行文件缺失。 | 先在终端直接运行命令，再修正客户端配置。 |
| 工具列表不可见 | server 启动失败或工具注册失败。 | 查看启动日志，确认官方 SDK stdio server 已运行并完成工具注册。 |
| `SESSION_BINDING_MISSING` | 当前会话没有默认实例且未传 `handle_id`。 | 先调用 `instance_use`，或在请求中显式传入 `handle_id`。 |
| `TX_QUEUE_FULL` | 发送速率超过底层写入速度。 | 降低发送频率、等待队列排空或清理 tx。 |
| `READ_TIMEOUT` | 设备暂未输出、超时太短或读取目标不匹配。 | 增加超时、确认设备响应或改用订阅。 |
| `SERIAL_PORT_BUSY` | 串口被其他实例或外部程序占用。 | 断开占用实例、关闭外部程序或换端口。 |
| 串口权限错误 | Linux/macOS 用户权限不足或驱动未授权。 | 调整设备组、udev 或系统隐私权限后重试。 |
| loopback 端口冲突 | 本地端口被其他进程占用。 | 使用动态端口或释放占用进程。 |
