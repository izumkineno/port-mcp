# 07 端口 MCP 错误模型与返回格式

本文档承接 [01-初版工程功能设计大纲](01-初版工程功能设计大纲.md)、[03-初版工具契约](03-初版工具契约.md)、[04-运行时架构设计](04-运行时架构设计.md)、[05-状态机与资源生命周期](05-状态机与资源生命周期.md) 与 [06-并发与缓冲区设计](06-并发与缓冲区设计.md)，固化初版工具返回外形、错误类别、细分错误码、调试信息、日志字段、脱敏规则和恢复建议。

本文档只定义工具调用结果和错误表达。具体测试矩阵由 `08-测试与验收计划.md` 承接；运行配置和 verbose 开关可由 `11-配置与运行说明.md` 承接。

## 设计结论

- 成功和失败都返回统一外形，顶层必须包含 `ok`、`tool`、`request_id`、`timestamp`；实例相关工具还必须包含 `handle_id` 和 `state`。
- 错误标识分两层：`error.category` 保持 [03-初版工具契约](03-初版工具契约.md) 中的稳定类别，`error.code` 提供可测试、可定位的细分错误码。
- 失败返回应尽可能包含 AI 自查需要的调试信息，例如当前状态、允许状态、资源摘要、参数名、限制值、任务名、底层错误 kind 和建议下一步。
- 工具返回默认脱敏敏感信息，不直接暴露系统用户名、完整本地路径、环境变量、原始长 payload、堆栈或未裁剪的底层错误文本。
- 内部日志可记录更完整的 cause chain、资源锁 key、任务退出原因和后台 close 结果，但日志字段也必须区分安全摘要和敏感细节。
- 运行时进入 `Error` 时，必须把可返回的错误摘要写入实例 `last_error`；后台清理失败、强制释放后台 close 失败只能写日志或调试事件。
- `ReadTimeout` 属于可预期业务结果，不改变实例状态；连接失败、后台读写致命失败和断开清理失败按 [05-状态机与资源生命周期](05-状态机与资源生命周期.md) 收敛到 `Error`。

## 统一成功返回

成功返回外形如下：

```json
{
  "ok": true,
  "tool": "port_send",
  "request_id": "req_20260525_000001",
  "timestamp": "2026-05-25T12:00:00.000Z",
  "handle_id": "h_tcp_001",
  "state": "Connected",
  "data": {
    "queued": true,
    "sent_bytes": 6
  },
  "warnings": []
}
```

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `ok` | 是 | 成功固定为 `true`。 |
| `tool` | 是 | 当前工具名。 |
| `request_id` | 是 | 单次工具调用 ID，用于关联日志。 |
| `timestamp` | 是 | 运行时生成的 UTC 时间。 |
| `handle_id` | 实例相关工具必填 | 请求或结果关联的实例句柄。 |
| `state` | 实例相关工具必填 | 操作完成后的实例状态。 |
| `data` | 是 | 工具特定结果。无业务数据时返回 `{}`。 |
| `warnings` | 是 | 非致命警告数组；无警告时返回空数组。 |

`instance_list`、`port_scan` 等全局工具可以省略 `handle_id` 和 `state`，但仍必须返回 `request_id`、`timestamp`、`data` 和 `warnings`。

## 统一失败返回

失败返回外形如下：

```json
{
  "ok": false,
  "tool": "port_connect",
  "request_id": "req_20260525_000002",
  "timestamp": "2026-05-25T12:00:03.000Z",
  "handle_id": "h_ser_002",
  "state": "Error",
  "error": {
    "category": "ResourceBusy",
    "code": "SERIAL_PORT_BUSY",
    "message": "Serial port COM3 is already owned by another connected instance.",
    "recovery_hint": "Disconnect the owning instance or choose another serial port.",
    "retryable": false,
    "details": {
      "resource": {
        "kind": "serial",
        "display": "COM3"
      },
      "owner_handle_id": "h_ser_001"
    }
  }
}
```

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `ok` | 是 | 失败固定为 `false`。 |
| `tool` | 是 | 当前工具名。 |
| `request_id` | 是 | 单次工具调用 ID，用于关联日志。 |
| `timestamp` | 是 | 运行时生成的 UTC 时间。 |
| `handle_id` | 请求包含时必填 | 请求中的句柄原样返回；句柄不存在也应返回。 |
| `state` | 能确定时必填 | 错误发生后的实例状态；句柄不存在时可省略。 |
| `error.category` | 是 | 稳定错误类别。 |
| `error.code` | 是 | 细分错误码。 |
| `error.message` | 是 | 面向 AI 和用户的可读消息。 |
| `error.recovery_hint` | 是 | 下一步恢复建议。 |
| `error.retryable` | 是 | 当前请求不改参数直接重试是否可能成功。 |
| `error.details` | 是 | 脱敏后的调试信息；无详情时返回 `{}`。 |

失败返回不得使用成功返回里的 `warnings` 承载错误。非致命问题应让工具成功并填充 `warnings`；阻塞当前工具语义的问题必须返回 `ok=false`。

## 错误类别与细分错误码

`category` 是跨版本稳定的高层类别；`code` 是实现、测试和恢复流程使用的细分原因。初版必须支持下表中的类别和建议错误码。

### `HandleNotFound`

| `code` | 场景 | 恢复建议 |
| --- | --- | --- |
| `HANDLE_NOT_FOUND` | `handle_id` 不存在。 | 调用 `instance_list` 获取当前实例。 |
| `HANDLE_RELEASED` | 句柄已释放，实例已移出表。 | 重新 `instance_create` 并配置连接。 |
| `SESSION_BINDING_MISSING` | 当前会话没有默认实例，且请求未提供 `handle_id`。 | 显式传入 `handle_id` 或调用 `instance_use`。 |

### `InvalidState`

| `code` | 场景 | 恢复建议 |
| --- | --- | --- |
| `STATE_NOT_ALLOWED` | 当前状态不允许调用该工具。 | 按 `details.allowed_states` 先执行配置、连接、断开或释放。 |
| `CONFIG_REQUIRED` | 未配置或配置不完整时连接。 | 调用 `serial_config` 或 `tcp_udp_config`。 |
| `CONNECTED_RELEASE_REQUIRES_FORCE` | `Connected` 状态普通释放被拒绝。 | 先 `port_disconnect`，或使用 `instance_release(force=true)`。 |
| `ERROR_REQUIRES_DISCONNECT` | `Error` 状态下直接重新配置或连接。 | 先 `instance_query`，再 `port_disconnect` 或释放重建。 |
| `DISCONNECTING_IN_PROGRESS` | 实例正在断开清理。 | 等待清理完成后查询状态。 |
| `SESSION_ID_UNAVAILABLE` | 当前 MCP SDK 或运行模式无法提供稳定会话 ID，但工具需要会话隔离。 | 若只是默认实例解析失败，显式传入 `handle_id` 后重试；若调用的是 `instance_use`、`port_subscribe_stream` 或 `port_unsubscribe_stream`，需启用稳定 session 或受控单会话开发模式后重试。 |
| `FEATURE_NOT_COMPILED` | 当前构建未启用某项可选能力，但请求已进入该能力路径，例如 `instance_create(type=Visa)`、`port_scan(type=Visa)` 或 `visa_config`。 | 使用启用对应 Cargo feature 的构建产物，或改用当前构建支持的实例类型。 |
| `VISA_RUNTIME_UNAVAILABLE` | 已编译 VISA feature，但运行环境缺少 VISA resource manager、底层 runtime 或驱动。 | 安装并验证本机 VISA runtime，或切换到无需 VISA 的实例类型。 |

### `InvalidArgument`

| `code` | 场景 | 恢复建议 |
| --- | --- | --- |
| `MISSING_REQUIRED_FIELD` | 缺少必填字段。 | 补齐 `details.field`。 |
| `INVALID_FIELD_TYPE` | 字段类型错误。 | 按 `details.expected_type` 修正。 |
| `INVALID_ENUM_VALUE` | 枚举值非法。 | 使用 `details.allowed_values` 中的值。 |
| `INVALID_RANGE` | 数值越界，例如端口号、超时或 `max_bytes`。 | 调整到 `details.min` 到 `details.max` 范围内。 |
| `TYPE_MISMATCH` | 工具与实例类型不匹配，例如对 TCP 实例调用串口配置。 | 使用匹配实例类型的配置工具。 |
| `INVALID_ADDRESS` | IP、主机名或监听地址格式非法。 | 修正地址格式。 |
| `SCAN_TARGET_NOT_ALLOWED` | 网络扫描目标不在允许范围内，例如非 loopback 地址。 | 改用 `127.0.0.1` 或 `::1`；非 loopback allowlist 属于后续进阶能力，初版不得通过隐藏配置启用。 |
| `INVALID_HEX` | Hex 数据含非法字符或奇数字符数。 | 使用偶数字符数量和合法十六进制字符。 |
| `TEXT_ENCODING_FAILED` | 文本按指定编码转字节失败。 | 改用 `UTF-8`、修正文案或使用 `hex`。 |
| `VISA_RESOURCE_NOT_FOUND` | VISA 资源地址不存在、不可解析，或当前 resource manager 无法识别该地址。 | 使用 `port_scan(type=Visa)` 获取可用地址，或修正 `resource_address`。 |

`port_scan` 的扫描错误判定顺序为：先校验目标是否为初版允许的 loopback 单主机；非 loopback、CIDR、通配地址、`0.0.0.0`、`::` 或 DNS 展开目标返回 `SCAN_TARGET_NOT_ALLOWED`。目标合法后再校验端口数量、端口范围和并发限制，超限返回 `SCAN_RANGE_TOO_LARGE`；结果数量或返回 payload 超限返回 `RESULT_TOO_LARGE`；端口号、超时等数值字段越界返回 `INVALID_RANGE`。

### `ResourceBusy`

| `code` | 场景 | 恢复建议 |
| --- | --- | --- |
| `SERIAL_PORT_BUSY` | 串口资源锁已被占用。 | 断开占用实例或换串口。 |
| `TCP_LISTEN_ADDR_BUSY` | TCP listen 地址端口冲突。 | 换监听地址端口或释放占用实例。 |
| `UDP_BIND_ADDR_BUSY` | UDP bind 地址端口冲突。 | 换绑定地址端口或释放占用实例。 |
| `VISA_RESOURCE_BUSY` | VISA 资源已被当前进程外或其他实例占用。 | 释放占用侧、关闭外部仪器工具，或改用其他 VISA 资源。 |
| `RESOURCE_CLOSING` | 同一资源的旧 transport 正在后台关闭，锁处于 `closing` tombstone。 | 稍后重试，或查看日志中的后台关闭结果。 |
| `RESOURCE_LOCK_STALE` | 运行时发现疑似残留资源锁。 | 查询实例列表，必要时释放相关实例后重试。 |

### `ConnectTimeout`

| `code` | 场景 | 恢复建议 |
| --- | --- | --- |
| `CONNECT_TIMEOUT` | TCP connect 或通用打开超时。 | 检查地址、设备和 `timeout_ms`。 |
| `SERIAL_OPEN_TIMEOUT` | 串口打开超时。 | 检查串口是否存在、权限和设备状态。 |
| `VISA_OPEN_TIMEOUT` | VISA open 会话超时。 | 检查 VISA 资源地址、底层 runtime、仪器电源和 `open_timeout_ms`。 |
| `SCAN_TIMEOUT` | 端口扫描单项或整体超时。 | 缩小范围或提高超时。 |

### `ReadTimeout`

| `code` | 场景 | 恢复建议 |
| --- | --- | --- |
| `READ_TIMEOUT` | `port_pull` 超时未读到数据。 | 重试、增加超时、确认设备输出或改用订阅。 |
| `NO_DATA_AVAILABLE` | 非等待模式下接收缓冲区为空。 | 稍后重试或订阅接收流。 |

### `WriteFailed`

| `code` | 场景 | 恢复建议 |
| --- | --- | --- |
| `WRITE_IO_FAILED` | 底层写入返回 I/O 错误。 | 查询 `last_error`，断开后重连。 |
| `READ_IO_FAILED` | 接收任务遇到不可恢复 I/O 错误。 | 查询错误，断开后重连。 |
| `TRANSPORT_CLOSED` | 底层连接已关闭或对端断开。 | `port_disconnect` 后重新连接。 |
| `TASK_FAILED` | 后台发送或接收任务异常退出。 | 查询任务名和原因，断开后重连。 |
| `DISCONNECT_FAILED` | 断开清理失败。 | 再次 `port_disconnect` 或 `instance_release(force=true)`。 |
| `RELEASE_FAILED` | 普通释放清理失败。 | 使用 `force=true` 或检查日志。 |
| `TX_QUEUE_CLOSED` | 发送入口已关闭。 | 查询状态，通常需要断开或重连。 |
| `VISA_ENUM_FAILED` | VISA resource manager 枚举资源失败。 | 检查 VISA runtime、过滤模式和本机驱动，再重试。 |
| `VISA_OPEN_FAILED` | VISA 资源 open 失败。 | 校验资源地址、资源占用情况和底层 runtime 状态。 |
| `VISA_WRITE_FAILED` | VISA 写入失败。 | 查询 `last_error`，检查终止符、超时和会话状态后重试。 |
| `VISA_READ_FAILED` | VISA 读取失败。 | 检查读超时、终止符和设备输出，再 `port_pull` 或重连。 |
| `VISA_QUERY_IDN_FAILED` | 可选 `*IDN?` 识别失败。 | 若连接已成功，可忽略识别失败并继续基础收发，或检查仪器是否支持 `*IDN?`。 |

### `BufferLimitExceeded`

| `code` | 场景 | 恢复建议 |
| --- | --- | --- |
| `TX_QUEUE_FULL` | 发送队列已满，新 `port_send` 被拒绝。 | 等待队列排空、降低发送速率或 `port_clear(tx)`。 |
| `TX_FRAME_TOO_LARGE` | 单次发送帧超过 `tx_frame_max_bytes`。 | 拆分数据或降低 payload 大小。 |
| `PULL_MAX_BYTES_EXCEEDED` | 请求的 `max_bytes` 超过 `pull_max_bytes`。 | 降低 `max_bytes`。 |
| `SUBSCRIBER_PAYLOAD_TOO_LARGE` | 订阅 `max_payload_bytes` 超过上限。 | 降低通知 payload 上限。 |
| `SUBSCRIBER_LIMIT_EXCEEDED` | 单实例订阅者数量超过 `max_subscribers_per_instance`。 | 取消不需要的订阅或提高受控配置上限。 |
| `SCAN_RANGE_TOO_LARGE` | 扫描范围或并发超过限制。 | 缩小扫描范围或降低并发。 |
| `RESULT_TOO_LARGE` | 工具结果超过返回上限。 | 降低请求范围或分批读取。 |

普通 `rx` 缓冲区满时按 [06-并发与缓冲区设计](06-并发与缓冲区设计.md) 丢旧保新并累计 `rx_dropped_bytes`，不直接返回 `BufferLimitExceeded`，除非当前工具请求本身超过限制。

## `error.details` 推荐字段

`error.details` 应围绕 AI 可恢复决策组织，字段允许按错误类型增减，但推荐使用下列稳定键名。

| 字段 | 适用场景 | 说明 |
| --- | --- | --- |
| `field` | 参数错误 | 出错字段名。 |
| `expected_type` | 类型错误 | 期望类型。 |
| `actual_type` | 类型错误 | 实际类型摘要。 |
| `allowed_values` | 枚举错误 | 可选值数组。 |
| `min` / `max` | 范围错误 | 允许范围。 |
| `actual` | 参数错误 | 脱敏和裁剪后的实际值。 |
| `current_state` | 状态错误 | 当前实例状态。 |
| `allowed_states` | 状态错误 | 当前工具允许状态。 |
| `next_actions` | 所有错误 | 建议工具调用序列。 |
| `resource` | 资源冲突、连接失败 | 脱敏资源摘要。 |
| `feature` | 可选能力未编译 | 触发错误所需的 feature 名，例如 `visa`。 |
| `owner_handle_id` | 资源冲突 | 占用资源的实例句柄，如可安全确定。 |
| `limit` | 容量错误 | 触发限制的阈值。 |
| `requested` | 容量错误 | 请求值。 |
| `queue_items` | 队列错误 | 当前队列长度。 |
| `buffer_bytes` | 缓冲区错误 | 当前缓冲区长度。 |
| `task` | 后台任务错误 | `receive`、`send`、`disconnect` 或 `background_close`。 |
| `io_kind` | 底层 I/O 错误 | 脱敏后的 I/O 错误类别，例如 `permission_denied`、`connection_reset`。 |
| `last_error_id` | 实例错误 | 关联 `last_error.error_id`。 |
| `backend` | 可选依赖错误 | 后端摘要，例如 `visa-rs` 或底层 resource manager。 |

`details` 不应塞入完整 payload。需要展示样例数据时，只返回裁剪后的 `preview`、`preview_encoding` 和 `omitted_bytes`。

## 脱敏规则

初版默认在工具返回中提供丰富但脱敏的调试信息。

| 信息类型 | 工具返回策略 | 日志策略 |
| --- | --- | --- |
| 串口名 | 可返回，例如 `COM3` 或 `/dev/ttyUSB0`。 | 可记录规范化资源锁 key。 |
| IP 和端口 | 可返回。 | 可记录规范化资源锁 key。 |
| 完整本地路径 | 默认只返回文件名或路径类别。 | 可记录完整路径，但标记为 sensitive。 |
| 系统用户名 | 不返回。 | 必须标记为 sensitive。 |
| 环境变量 | 不返回变量值。 | 默认不记录值；如必须记录则标记 sensitive。 |
| 原始 payload | 只返回短 preview 和长度。 | 可按调试配置采样，默认裁剪。 |
| 底层 OS 错误文本 | 返回脱敏摘要和 `io_kind`。 | 可记录完整 cause chain。 |
| panic 或堆栈 | 不返回。 | 可记录，必须标记 sensitive。 |

如果未来增加 `verbose_error_details`，默认仍应关闭。即使开启，也不得返回环境变量值、认证信息或未裁剪的大 payload。

## 日志事件格式

每次工具失败、状态进入 `Error`、后台任务异常退出、资源锁异常或后台 close 失败，都应写入结构化日志事件。

```json
{
  "event": "tool_error",
  "timestamp": "2026-05-25T12:00:03.000Z",
  "request_id": "req_20260525_000002",
  "tool": "port_connect",
  "handle_id": "h_ser_002",
  "category": "ResourceBusy",
  "code": "SERIAL_PORT_BUSY",
  "state_before": "Configured",
  "state_after": "Error",
  "resource_key": "serial:COM3",
  "message": "Serial port busy",
  "cause_chain": [
    { "kind": "resource_lock", "message": "owned by h_ser_001" }
  ],
  "sensitive": false
}
```

| 字段 | 说明 |
| --- | --- |
| `event` | `tool_error`、`state_error`、`task_error`、`resource_lock_error`、`background_close_error` 等。 |
| `timestamp` | UTC 时间。 |
| `request_id` | 关联工具调用；后台事件可为空。 |
| `tool` | 关联工具；后台事件可为空。 |
| `handle_id` | 关联实例；释放后后台事件也可以记录旧句柄。 |
| `category` / `code` | 错误类别和细分错误码。 |
| `state_before` / `state_after` | 状态变化前后。 |
| `resource_key` | 内部规范化资源 key。 |
| `message` | 脱敏摘要。 |
| `cause_chain` | 内部原因链，按日志配置裁剪。 |
| `sensitive` | 是否包含敏感信息。 |

日志不是工具返回的一部分，但 `request_id` 和 `last_error.error_id` 应能把工具返回、实例查询和内部日志串起来。

## `last_error` 摘要

`instance_query` 的 `data.last_error` 用于让 AI 在 `Error` 状态下恢复上下文。建议格式如下：

```json
{
  "error_id": "err_20260525_000007",
  "at": "2026-05-25T12:00:03.000Z",
  "tool": "port_connect",
  "category": "ResourceBusy",
  "code": "SERIAL_PORT_BUSY",
  "message": "Serial port COM3 is already owned by another connected instance.",
  "recovery_hint": "Disconnect the owning instance or choose another serial port.",
  "details": {
    "resource": { "kind": "serial", "display": "COM3" },
    "owner_handle_id": "h_ser_001"
  }
}
```

`last_error` 必须脱敏，不能直接暴露内部堆栈。新错误覆盖旧 `last_error`；成功配置可清理配置相关 `last_error`，但普通 `port_clear` 不应重置错误历史。

## 状态影响规则

| 错误场景 | 返回类别 | 实例状态影响 |
| --- | --- | --- |
| 参数校验失败 | `InvalidArgument` | 不改变状态。 |
| 句柄不存在 | `HandleNotFound` | 无实例可变更。 |
| 工具状态不允许 | `InvalidState` | 不改变状态。 |
| 可选 feature 未编译 | `InvalidState` | 不改变状态。 |
| 可选 runtime 缺失 | `InvalidState` | 对尚未连通的实例保持 `Created` 或 `Configured`；若在连接流程中探测到不可用，可进入 `Error` 并写入 `last_error`。 |
| 资源锁占用 | `ResourceBusy` | `port_connect` 进入 `Error`。 |
| 连接或打开超时 | `ConnectTimeout` | `port_connect` 进入 `Error`。 |
| `port_pull` 无数据超时 | `ReadTimeout` | 不改变状态。 |
| 底层读写致命失败 | `WriteFailed` | 进入 `Error`。 |
| 断开清理失败 | `WriteFailed` | 进入或保持 `Error`。 |
| 发送队列满 | `BufferLimitExceeded` | 不改变状态。 |
| 单帧、拉取或订阅上限超限 | `BufferLimitExceeded` | 不改变状态。 |
| 普通 `rx` 缓冲区溢出 | 无工具失败 | 不改变状态，累计统计。 |

状态提交、资源锁回滚和错误态恢复以 [05-状态机与资源生命周期](05-状态机与资源生命周期.md) 为准。

## 警告返回

警告表示工具已经成功，但有非阻塞信息需要 AI 注意。例如通知 payload 被截断、扫描结果被限制、断开时丢弃了未发送队列项。

```json
{
  "code": "SUBSCRIBER_NOTIFICATION_TRUNCATED",
  "message": "Notification payload was truncated to subscriber_payload_max_bytes.",
  "details": {
    "original_bytes": 32768,
    "payload_bytes": 16384
  }
}
```

| `code` | 场景 |
| --- | --- |
| `RX_BYTES_DROPPED` | 普通 `rx` 缓冲区曾发生丢旧保新。 |
| `TX_ITEMS_DROPPED_ON_CLEAR` | `port_clear(tx|all)` 丢弃了待发送项。 |
| `TX_ITEMS_DROPPED_ON_DISCONNECT` | 断开时丢弃了未写入项。 |
| `SUBSCRIBER_NOTIFICATION_TRUNCATED` | 订阅通知 payload 被截断。 |
| `SCAN_RESULTS_TRUNCATED` | 扫描结果达到返回上限。 |

警告不改变 `ok=true`。如果警告代表的情况使当前工具无法满足语义，应改为失败返回。

## 恢复建议规范

`recovery_hint` 应直接告诉 AI 下一步可调用什么工具，而不是泛泛描述。

| 错误类别 | 建议写法 |
| --- | --- |
| `HandleNotFound` | `Call instance_list and choose an existing handle_id, or create a new instance.` |
| `InvalidState` | `Call instance_query, then run serial_config/tcp_udp_config, port_connect, or port_disconnect according to allowed_states.` |
| `InvalidArgument` | `Fix the field named in details.field and retry the same tool.` |
| `ResourceBusy` | `Disconnect the owning instance or choose another resource.` |
| `ConnectTimeout` | `Check the device/address, increase timeout_ms, then retry port_connect.` |
| `ReadTimeout` | `Retry port_pull, increase timeout_ms, or use port_subscribe_stream.` |
| `WriteFailed` | `Call instance_query, then port_disconnect and reconnect if needed.` |
| `BufferLimitExceeded` | `Lower the requested size, wait for buffers to drain, or clear the relevant buffer.` |

中文用户可读消息可后续本地化；初版内部错误类别、错误码和字段名统一使用英文，避免代码和测试出现多语言分歧。

## 设计验收标准

本文档完成后，初版工程应能据此实现和测试：

- 所有工具成功返回都包含统一 `ok=true` 外形，失败返回都包含统一 `ok=false` 外形。
- 每个失败都有稳定 `category` 和细分 `code`，且 `code` 能归入 [03-初版工具契约](03-初版工具契约.md) 的错误类别。
- 错误返回提供足够 AI 自查的信息，包括当前状态、允许状态、资源摘要、限制值或参数字段。
- 工具返回默认脱敏敏感信息，不泄漏系统用户名、完整本地路径、环境变量值、未裁剪 payload 或堆栈。
- 会导致实例进入 `Error` 的错误必须写入 `last_error`，并能通过 `instance_query` 查看摘要。
- `ReadTimeout`、发送队列满、参数错误和状态错误不应误改实例状态。
- 日志事件能通过 `request_id` 或 `last_error.error_id` 与工具返回关联。
