# 25 进阶能力施工拆分与验收门槛

本文档承接 [02-进阶实现大纲](02-进阶实现大纲.md)、[13-代码施工大纲](13-代码施工大纲.md)、[17-初版施工任务拆分清单](17-初版施工任务拆分清单.md) 与 [24-MCP工具列表与调用收发详解](24-MCP工具列表与调用收发详解.md)，把进阶能力总纲继续细分为可施工、可验收、可排期的落地文档。

本文档不逐字段定义新的 MCP 工具参数或返回 JSON，不下钻到代码级函数和文件清单，也不重新拆分初版已经完成的能力。它只覆盖 [02-进阶实现大纲](02-进阶实现大纲.md) 中的进阶范围，并按能力章节给出推荐施工切片、主要模块触点、验收门槛和不做事项。

## 设计结论

- 进阶能力应继续沿用初版的运行时、资源锁、发送队列、接收缓冲区、订阅广播、错误模型和脱敏规则，不允许绕过既有主链路另起一套运行时。
- 进阶能力按 `02` 的能力章节逐块细分，而不是先按里程碑重排；这样后续可以按能力块单独立项、延后或并行。
- 可以增加少量直接服务于进阶落地或调试闭环的辅助工具，但不能把范围扩成泛化设备平台、资产管理系统或网络拓扑发现平台。
- 任何新增工具都必须继承初版的容量限制、权限边界、审计要求和结构化错误返回，不因“只是 helper”而跳过这些约束。
- 进阶阶段的验收仍然遵循“自动化优先、真实硬件兜底、证据先于结论”的原则；VISA 和真实设备能力继续使用可选依赖和手工验收收口。

## 适用范围与非目标

| 范围 | 是否覆盖 | 说明 |
| --- | --- | --- |
| VISA / USBTMC / GPIB 仪器能力 | 覆盖 | 作为进阶设备接入能力。 |
| 协议辅助工具 | 覆盖 | 包括字符串/Hex、Modbus、SCPI、AT、SLIP 和受控辅助探测工具。 |
| TCP/UDP 传输增强 | 覆盖 | 包括多客户端 listen、UDP datagram 元数据和多来源管理。 |
| 高级流式订阅 | 覆盖 | 包括按行、正则、Hex 匹配、组帧和批量聚合。 |
| 链路维持与自动重连 | 覆盖 | 包括心跳、退避、重连状态和统计。 |
| 异常注入 | 覆盖 | 作为容错与调试增强能力。 |
| 数据安全、性能治理、调试导出 | 覆盖 | 作为所有进阶能力的横向底座。 |
| 新工具逐字段 JSON 契约 | 不覆盖 | 详细参数和返回格式应在后续专门契约文档中定义。 |
| 代码级函数/文件清单 | 不覆盖 | 这里只到模块和运行时触点层。 |
| 初版能力重拆 | 不覆盖 | 以 01、03、13、17、24 为准。 |

## 统一施工原则

### 1. 兼容性原则

- 进阶能力必须默认保持初版兼容行为；新增模式、触发条件或扩展返回不得破坏初版现有调用路径。
- 无法在默认行为下保持兼容时，应通过显式参数、feature、可选依赖或新增工具隔离，而不是偷偷改旧工具语义。

### 2. 运行时原则

- 不新增第二套实例表、资源锁、发送队列或订阅系统。
- 新增后台任务必须挂接到现有实例生命周期，在 `disconnect`、`release`、`force release` 和 `Error` 收敛路径中可被统一停止。
- 所有设备、网络和协议 helper 输出都要能回到 `port_send` / `port_pull` / 订阅路径，不做平行收发通道。

### 3. 安全与治理原则

- 新 helper 不能绕过 payload 大小限制、通知限速、脱敏规则和错误结构。
- 网络侧增强必须继续遵守初版的最小暴露原则；即便允许适度扩展扫描或探测，也必须保持默认关闭、显式允许和审计留痕。
- 调试导出和异常注入只服务于排查与复现，不允许变成隐式侧信道。

### 4. 文档与验收原则

- 新能力进入施工前，至少要有能力边界、验收门槛和不做事项。
- 每一块能力都应能独立形成施工记录或 acceptance 记录，而不是只在总纲中口头通过。
- 当 helper 工具开始稳定到影响对外调用时，再拆专门契约文档，不在本文提前固化字段细节。

## 能力 1：VISA 仪器控制

### 施工目标

在不污染基础构建的前提下，为 VISA、USBTMC、GPIB 等仪器资源提供受控枚举、配置和 SCPI 一问一答能力，并让缺依赖场景保持可诊断退化。

### 推荐能力切片

| 切片 | 目标结果 | 主要触点 | 备注 |
| --- | --- | --- | --- |
| V1 资源枚举 | 可列出本机 VISA 资源摘要 | `transport`、`app::port_service`、`model` | 需要与串口/TCP/UDP 扫描结果隔离。 |
| V2 实例配置 | `visa_config` 可写入地址和超时 | `app::config_service`、`model::config`、`runtime` | 不在配置阶段强依赖真实连通性。 |
| V3 仪器识别 | 可选 `*IDN?` 识别并返回摘要 | `transport::visa`、`runtime::tasks`、`model::data` | 识别失败不应阻塞保存配置。 |
| V4 基础收发 | 通过既有 `port_send` / `port_pull` 完成 SCPI 往返 | `transport::visa`、`runtime::tasks`、`mcp` | 不新增平行发送 API。 |
| V5 缺依赖退化 | 无 VISA runtime 时返回稳定错误 | `model::error`、`app`、`mcp` | 不能影响串口和网络能力。 |

### 验收门槛

- 可枚举至少一类 VISA 资源地址。
- 有模拟器或真实设备时，可完成一次 SCPI 一问一答。
- `visa_config` 与 `instance_query` 能表达配置和识别摘要。
- 缺少底层依赖、驱动或设备时，返回清晰错误且不污染其他类型实例。

### 不做事项

- 不把 VISA 能力做成基础构建硬依赖。
- 不在第一版承诺全厂商兼容。
- 不在这里扩展复杂波形抓取、截图、文件系统访问等高阶仪器特性。

## 能力 2：协议辅助工具

### 施工目标

把高频协议处理中的重复、易错、可纯计算的部分抽成稳定 helper，并确保输出仍然服从既有发送、拉取、订阅和脱敏链路。

### 2.1 字符串与十六进制转换

| 切片 | 目标结果 | 主要触点 | 备注 |
| --- | --- | --- | --- |
| P1 基础转换 | `str_to_hex`、`hex_to_str` 可稳定处理常见编码 | `model::data`、`util`、`mcp` | 无状态纯 CPU 工具。 |
| P2 错误细化 | 非法 hex、不可解码字节、超长输入返回稳定错误 | `model::error`、`mcp::response` | 错误不能只给“解析失败”。 |
| P3 审计摘要 | helper 输入输出可按摘要进入调试导出 | `model::data`、`docs/12` | 不保留不必要原始敏感数据。 |

### 2.2 Modbus helper

| 切片 | 目标结果 | 主要触点 | 备注 |
| --- | --- | --- | --- |
| P4 RTU pack | 生成常见功能码请求帧 | `util`、`model::data`、`mcp` | 先做 RTU CRC16。 |
| P5 RTU unpack | 可校验和解析响应帧 | `util`、`model::error` | 明确区分 CRC 错误和协议字段错误。 |
| P6 ASCII 扩展占位 | 预留 LRC 和 ASCII 能力 | `docs`、`model` | 第一版可只留边界，不必须实现。 |

### 2.3 SCPI / AT / SLIP helper

| 切片 | 目标结果 | 主要触点 | 备注 |
| --- | --- | --- | --- |
| P7 SCPI helper | 规范化命令文本和响应摘要 | `util`、`model::data` | 不替代真实仪器手册。 |
| P8 AT helper | 规范化 AT 指令文本与基础响应分类 | `util`、`model::error` | 不硬编码大量厂商私有扩展。 |
| P9 SLIP helper | 提供 encode/decode 和非法转义检测 | `util`、`model::data` | 输出仍应可直接喂给 `port_send`。 |

### 2.4 可允许的辅助工具扩展

用户允许适度补充与主线直接相关的辅助工具。建议只纳入以下边界内的工具：

| 辅助工具方向 | 目标结果 | 边界 |
| --- | --- | --- |
| `device_port_probe` 类工具 | 在候选串口、候选 TCP 端口或受控地址集合上并发发送探测请求，定位目标设备可能挂载的多个端口 | 只允许有限候选集合、受控并发、受控超时和审计记录；不是通用扫描器。 |
| `probe_profile` 摘要 | 复用固定的请求 payload、期待响应或特征匹配规则 | 只表达探测模板，不扩展成设备指纹库平台。 |
| `response_matcher` 摘要 | 用于探测阶段的最小匹配规则，例如固定字节、文本片段或 checksum 通过 | 只服务于定位与确认，不替代完整协议解析。 |

### 验收门槛

- helper 输出可直接服务 `port_send`、`port_pull` 或调试导出。
- 对非法输入、超长输入和不可解析响应返回稳定错误。
- `device_port_probe` 类工具必须有受控候选集、并发限制、总超时和审计记录。
- helper 不绕过实例发送队列、容量限制和脱敏规则。

### 不做事项

- 不把 helper 做成完整设备管理平台。
- 不引入与 02 主线弱相关的资产管理、设备指纹库和拓扑发现能力。
- 不在本文固定 helper 的最终 JSON 契约。

## 能力 3：TCP/UDP 传输增强

### 施工目标

在保持初版兼容的前提下，补足多客户端 TCP listen、UDP datagram 元数据和多来源管理，使网络调试从“最低可用”提升到“可用于复杂现场”。

### 推荐能力切片

| 切片 | 目标结果 | 主要触点 | 备注 |
| --- | --- | --- | --- |
| N1 TCP 多客户端 listen | 一个 listen 实例可跟踪多个 peer | `transport::tcp`、`runtime::instance`、`runtime::stats` | 需要 peer id 或子句柄语义。 |
| N2 TCP 发送目标策略 | 可指定目标 peer 或广播 | `app::port_service`、`model::data`、`mcp` | 不能破坏单客户端默认行为。 |
| N3 UDP datagram 元数据 | 可查询来源地址、接收时间和长度 | `transport::udp`、`runtime::buffers`、`model::data` | 不应破坏普通 `port_pull` 的字节流语义。 |
| N4 UDP 多来源管理 | 支持最近来源、过滤、统计和来源视图 | `runtime::stats`、`app`、`mcp` | 适合多设备回包场景。 |
| N5 调试可见性 | peer/source 维度的错误和日志可观测 | `model::error`、`docs/12`、`24` 后续续写 | 需要和调试导出联动。 |

### 验收门槛

- 多客户端 listen 场景下，连接、发送、断开和统计彼此可区分。
- UDP datagram 元数据可订阅、可查询，且初版字节流读取语义保持兼容。
- peer/source 级别的资源锁、错误、调试记录和导出可用。

### 不做事项

- 不在第一版引入复杂路由脚本引擎。
- 不把 UDP 改造成必须保留 datagram 边界的全新 pull 契约。
- 不把多客户端场景做成新的独立实例体系。

## 能力 4：高级流式订阅

### 施工目标

把初版 `on_receive` 广播扩展成更适合文本协议、高频流和二进制组帧场景的触发体系，同时保持背压、限速和隔离可控。

### 推荐能力切片

| 切片 | 目标结果 | 主要触点 | 备注 |
| --- | --- | --- | --- |
| S1 `line` 触发 | 支持 `\\n` / `\\r\\n` 成行推送 | `runtime::subscription`、`runtime::buffers` | 适合日志和 AT 文本。 |
| S2 `batch` 触发 | 支持时间窗或字节窗聚合 | `runtime::subscription`、`runtime::stats` | 优先解决通知风暴。 |
| S3 `match_hex` 触发 | 匹配固定字节序列 | `runtime::subscription`、`model::data` | 服务二进制协议。 |
| S4 `match_regex` 触发 | 支持受限正则匹配 | `runtime::subscription`、`util`、`model::error` | 需要复杂度和超时保护。 |
| S5 `frame_hex` 组帧 | 基于帧头、长度、校验组装帧 | `runtime::subscription`、`model::data` | 要考虑半包和坏帧。 |

### 验收门槛

- 多订阅者可以在同一实例上使用不同触发条件。
- 高频输入场景下不会出现无界通知风暴。
- 正则、组帧和 Hex 匹配都有输入长度、计算复杂度和超时边界。
- 二进制流在通知和导出中可保留 Hex 表达，不因文本编码损坏。

### 不做事项

- 不允许任意复杂、无超时保护的正则执行。
- 不把高级订阅做成替代普通 `port_pull` 的唯一读取方式。
- 不隐式改变初版 `on_receive` 的默认订阅行为。

## 能力 5：链路维持与自动重连

### 施工目标

补足长连接调试中的心跳、退避和重连闭环，让连接维持行为可以配置、统计、诊断和安全停止。

### 推荐能力切片

| 切片 | 目标结果 | 主要触点 | 备注 |
| --- | --- | --- | --- |
| K1 心跳配置 | `keep_alive_config` 可启停和更新 | `runtime::tasks`、`app`、`model::config` | 心跳必须走同一发送队列。 |
| K2 心跳统计 | 心跳计数与用户发送计数分离 | `runtime::stats`、`instance_query` | 避免污染人工调试统计。 |
| K3 重连策略 | `reconnect_config` 表达退避、上限、抖动 | `runtime::tasks`、`runtime::instance` | 必须可停、可查。 |
| K4 重连诊断 | 重连次数、最后成功、最后失败和停止原因可见 | `runtime::stats`、`model::error`、`docs/12` | 方便调试闭环。 |
| K5 用户动作优先 | 主动断开或释放时立即停止心跳/重连 | `runtime`、`app` | 不允许“用户已断开但后台还在重连”。 |

### 验收门槛

- 心跳可启用、更新、停止，并进入既有发送队列。
- 自动重连按退避和上限运行，到达上限后进入可诊断状态。
- 用户主动断开或释放实例时，相关后台任务全部停止。
- 统计和订阅不会因为重连而产生重复、混乱或无法解释的数据。

### 不做事项

- 不做无限重连。
- 不在第一版引入复杂策略脚本或条件表达式。
- 不让心跳和重连拥有绕过用户发送约束的特殊通道。

## 能力 6：异常注入

### 施工目标

让开发者可以在可控范围内模拟断连、延迟、丢包和校验错误，以复现设备容错和现场问题。

### 推荐能力切片

| 切片 | 目标结果 | 主要触点 | 备注 |
| --- | --- | --- | --- |
| F1 注入配置 | `inject_error` 表达异常类型和参数 | `app`、`runtime::instance`、`model::config` | 应支持查询和清理。 |
| F2 发送前改写 | `corrupt_tx` / `bad_checksum` 生效可观测 | `runtime::tasks`、`model::data` | 要保留注入标记。 |
| F3 接收侧扰动 | `drop_rx` / `delay` / `disconnect` 可复现 | `runtime::tasks`、`runtime::buffers` | 不能污染其他实例。 |
| F4 调试记录联动 | 受注入影响的数据和错误能被标记 | `docs/12`、`model::data`、导出能力 | 否则复盘价值有限。 |

### 验收门槛

- 注入状态可启用、查询、停止和清理。
- 注入行为只影响目标实例。
- 调试记录和导出材料能明确标注受注入影响的数据或错误。

### 不做事项

- 不实现会跨实例传播的全局故障模式。
- 不把异常注入隐藏成普通错误。
- 不把异常注入做成远程攻击面或默认开启能力。

## 能力 7：数据安全、性能治理与调试导出

### 施工目标

把进阶能力引入后的复杂性收束到可治理、可限速、可导出和可脱敏的工程边界内，避免 helper、订阅增强和网络探测把系统推回不可控状态。

### 推荐能力切片

| 切片 | 目标结果 | 主要触点 | 备注 |
| --- | --- | --- | --- |
| G1 输入与 payload 限制 | helper、组帧、探测工具都有大小上限 | `util::limits`、`model::error` | 防止 CPU 和内存被单请求放大。 |
| G2 会话与实例配额 | 针对通知、探测、批量任务加入更细粒度配额 | `runtime::stats`、`runtime::subscription` | 进阶能力更容易引发风暴。 |
| G3 可控扫描/探测 | allowlist、dry-run、审计日志和每目标限速可用 | `app::port_service`、`docs/11`、`docs/12` | 保持默认关闭。 |
| G4 日志滚动与采样 | 长连接和高频流场景下避免日志无限增长 | `util`、`docs/11`、`docs/12` | 要和导出格式兼容。 |
| G5 调试导出 | 导出会话记录、收发摘要、错误链和复现实验脚本 | `docs/12`、`model::data`、`mcp` 后续扩展 | 输出必须默认脱敏。 |

### 验收门槛

- 新 helper、探测和订阅能力都有明确大小、频率、并发和总量上限。
- 受控扫描或探测默认关闭，并要求显式配置和审计留痕。
- 调试导出能输出脱敏包，并能区分脱敏包与敏感原始材料的处理路径。
- 长时间运行下日志和缓冲不会无界增长。

### 不做事项

- 不通过隐藏环境变量偷偷启用广域扫描。
- 不输出未脱敏原始 payload、凭据或完整本地路径。
- 不把调试导出做成默认无限留存的归档系统。

## 横向模块触点总表

| 模块层 | 进阶阶段主要责任 |
| --- | --- |
| `model` | 扩展实例配置、统计、错误码、订阅触发摘要、导出摘要和注入状态表达。 |
| `util` | 承接纯计算 helper、编码转换、校验算法、受限匹配和限制值工具。 |
| `runtime` | 继续作为任务、缓冲、订阅、统计、配额、重连和异常注入的唯一运行时入口。 |
| `transport` | 负责 VISA、增强 TCP/UDP 语义和真实 I/O 细节，不拥有工具可见契约。 |
| `app` | 编排新 helper、探测、配置和运行时策略切换。 |
| `mcp` | 注册新工具、包装统一返回、保留 request/session 诊断和脱敏边界。 |
| `docs` | 固化新增契约、施工记录、验收记录和调试导出格式。 |

## 推荐实施顺序

本文主体按能力章节组织，但建议的落地顺序仍可分波次推进：

1. 协议辅助工具基础层  
先做 `str_to_hex`、`hex_to_str`、Modbus RTU pack/unpack，以及最小的 SCPI/AT/SLIP helper，尽早降低人工手写帧错误率。

2. 高级订阅的低风险切片  
先做 `line` 和 `batch`，因为它们与初版 `on_receive` 广播最接近，也最能缓解通知风暴和文本调试痛点。

3. 链路维持  
先接入心跳，再接入自动重连，确保后台任务停止规则和统计口径先稳定。

4. TCP/UDP 传输增强  
在初版网络能力稳定的基础上，加入多客户端 listen、UDP datagram 元数据和多来源管理。

5. 调试闭环辅助工具  
纳入 `device_port_probe` 一类受控并发探测工具，以及调试导出能力，服务复杂现场定位。

6. VISA  
作为可选依赖能力后置接入，避免阻塞基础构建和普通开发者环境。

7. 异常注入  
在连接、订阅、统计和导出都足够稳定后加入，否则很难区分注入问题与基础运行时缺陷。
